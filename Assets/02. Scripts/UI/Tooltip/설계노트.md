# TooltipPopup — 설계 결정과 기반 지식

"왜 이렇게 만들었나"와 "그 판단에 필요한 UI Toolkit 배경지식"을 남긴다. 사용법은 [README.md](README.md).

---

## 1. 출발: 무엇이 문제였나

툴팁이 **4개 구현으로 파편화**돼 있었다.

| 구현 | 부착 방식 | 문제 |
|---|---|---|
| `EquipSlotsPresenter` (장비) | 요소 기준, 문서 루트에 부착 | **clamp 없음 → 화면 이탈** |
| `SkillTooltipView` (스킬) | 마우스 좌표, 자체 UIDocument | clamp을 280×150 **고정 추정** + 색 하드코딩 |
| `BuildingTooltipView` (건물) | 마우스 좌표, 자체 UIDocument | clamp을 300×220 **고정 추정** + 색 하드코딩 |
| 트레잇 태그 | Unity 내장 `tag.tooltip` | **런타임 미렌더 → 사실상 툴팁 없음**(에디터 전용 기능, Unity 6.3 UIDocument서 안 뜸) |

공통 핵심 버그: **clamp을 고정 크기로 추정**하거나 아예 없어서, 내용이 추정보다 길면 또 잘렸다. 같은 "컨테이너+테두리+위치계산"이 3벌 복붙돼 있었다.

## 2. 기반 지식 (UI Toolkit)

이번 설계 판단에 반복적으로 쓰인 사실들.

### 2.1 Panel(패널)이란
- `PanelSettings` 에셋이 런타임 **Panel**을 만든다. **같은 PanelSettings를 쓰는 모든 `UIDocument`는 같은 Panel(같은 `visualTree`)에 그려진다.**
- `panel.visualTree` = 패널 최상위 요소. 각 `UIDocument.rootVisualElement`는 그 **아래 자식**으로 붙는다. `UIDocument.sortingOrder`는 *한 패널 안에서* 문서들 간 순서일 뿐.
- **Bond**: 게임용 PanelSettings 2개 — 메인(`UI Toolkit/PanelSettings`, Town/Battle 등 대부분)과 Juno(`JUNO_PanelTest`, 맵/이벤트). **CharacterDetail·CombatPanel·EquipSlots는 메인 하나를 공유.** "백드롭 sortOrder -10"은 별도 패널이 아니라 같은 패널 안 UIDocument 순서다.

### 2.2 좌표계 (panel space)
- `worldBound` / `WorldToLocal`은 **panel space** 기준이다. 메인 패널은 `ScaleWithScreenSize`, 기준 해상도 **1200×800, match width** → panel space 폭은 **항상 1200**(실제 픽셀이 아님). 1200은 PanelSettings의 Reference Resolution(임의로 정한 디자인 캔버스값)이고, match width라 높이는 화면비로 결정된다(16:9면 ≈675).
- **함의**: `Screen.width/height`로 clamp하면 비원본 해상도에서 어긋난다(요소 bound는 panel space인데 Screen은 실제 픽셀). → `panel.visualTree.worldBound`(같은 panel space)로 잡아야 정확. 기존 스킬/건물 툴팁의 잠재 버그이기도 하다.

### 2.3 USS 캐스케이드 (투명 버그의 핵심)
- 스타일시트는 **붙은 요소의 하위로만** 흐른다. UXML `<Style src=...>`는 그 **문서 루트**에 붙는다(예: `EquipSlots.uss`는 `CharacterCombatPanel.uxml` / `CharacterDetail.uxml` 루트에 `<Style>`로). → 문서 루트 하위에만 적용.
- 단, **테마 변수**(`--bg-deep` 등)는 `PanelSettings.themeStyleSheet`로 패널 전체(`:root`)에 적용 → 어디서든 상속.
- **함정**: `panel.visualTree`(패널 최상위)는 문서 루트보다 **위**라, 문서 `<Style>`의 클래스 규칙(`equip-slots__tooltip`)이 **안 닿는다.** 변수는 닿지만 그 변수를 *쓰는 규칙*이 없으면 무의미 → **배경 투명.**

### 2.4 인라인 vs USS 클래스
- **인라인**(`el.style.x = ...`): 요소에 값이 박혀 **위치 무관·항상 최우선.** 단 `var()` 테마 변수를 **못 쓴다**(색 하드코딩 필요).
- **USS 클래스**: 테마 변수 사용 가능·재사용, 단 **캐스케이드 의존**(마운트 위치가 중요).

### 2.5 레이아웃 측정 타이밍 / 렌더 시점
- 콘텐츠를 막 넣은 직후엔 크기 미정. `display: None → Flex` 후 레이아웃 패스가 돌면 `GeometryChangedEvent`가 발생 → **그때** 실제 크기로 위치를 계산한다("추정"이 아닌 "실측").
- **렌더는 이벤트 핸들러가 끝난 뒤** 일어난다. 한 핸들러 안에서 `visibility`를 Hidden→Visible로 바꾸면 **중간 Hidden은 화면에 안 그려진다.** `Mount`가 Hidden을 set해도 이어지는 `Reposition`이 같은 호출에서 Visible로 덮으니, 마우스 추종(move마다 재마운트)에서도 **깜빡임이 없다.**
- `element.layout`은 마지막 레이아웃 패스 값을 **유지**한다(계층을 바꿔도 즉시 0이 되지 않음). 그래서 추종 2번째 프레임부터는 `w>0`이라 `Reposition`이 바로 통과 → `w<=0`(레이아웃 전) early-return은 **첫 표시 1프레임만** 걸린다.

### 2.6 특이도 / 적용 순서 (덮어쓰기 우선)
- 인라인 > USS 규칙. USS끼리는 특이도(클래스 수 등)로, **동률이면 더 깊은(요소에 가까운) 시트가 나중에 적용돼 이긴다.** → 문서 USS의 `.tooltip--x`가 패널 루트의 기본 `.tooltip`을 자연히 덮는다.

## 3. 결정들

### A. 위치만 통합, 스타일은 호출부 소유
스킬/건물 툴팁의 **현재 룩이 괜찮고**, 스타일가이드에 **툴팁 규정이 없어** 스타일 통일은 별도 논의가 필요하다. 그래서 엔진은 임의 `VisualElement`를 받아 **배치만** 한다. 화면이탈(진짜 문제)만 한 곳에서 해결하고, 비주얼은 안 건드린다.

### B. 정적 엔진 (DI 아님)
어디서나 ceremony(사전 설정, 스코프 등록 같은) 없이 호출 가능해야 했다(비-DI MonoBehaviour 포함). 상태(레이어)는 **정적 필드가 아니라 패널 트리에 심어** 보관한다 → "정적 가변 싱글톤 금지" 관례를 위반하지 않는다.

### C. 마운트 위치 — 최상위 시도 → 문서 루트로 정정 ★핵심
- **처음**: `panel.visualTree`(패널 최상위)에 부착. 이유 = ① 모든 것 위에 그리기 ② `overflow:hidden` 클리핑 회피 ③ 전체화면 clamp.
- **문제**: 문서 USS 캐스케이드 **밖**이라 `equip-slots__tooltip` 배경이 투명(§2.3).
- **재검토**: 그 3가지가 최상위를 *요구*하지 않는다 —
  - ① 문서 루트에서도 `BringToFront`면 그 문서 안 최상단.
  - ② 문서 루트는 보통 전체 크기라 중간 컨테이너에 안 잘린다(툴팁은 그 컨테이너들의 형제 가지).
  - ③ clamp은 부모와 무관하게 그냥 화면 전체 기준으로 계산하면 됨.
  - 결정적으로 **원래 장비 툴팁이 이미 문서 루트(`_root`)에 붙여 멀쩡했고 clamp만 없었다.** 최상위로 끌어올린 게 오히려 투명을 만든 것.
- **결정**: **anchor의 문서 루트**에 마운트(B 안). `ResolveDocumentRoot`가 anchor → 패널루트 직속 자식(= UIDocument 루트)까지 올라가 찾는다. 투명 **근본 해결** + 호출부 테마클래스 덮어쓰기 가능(캐스케이드 안).
- **트레이드오프**: "자기 문서 안에서만" 최상단. `sortingOrder`가 더 높은 다른 문서가 덮을 수 있음 — **현재 프로젝트엔 해당 없음**(백드롭은 뒤 -10, Detail/Combat이 앞). 진짜 필요해지면 그 케이스만 오버레이로 빼면 됨.

### D. 스타일 모델 = 기본 + 덮어쓰기
기본 `.tooltip`은 **패널 루트에 주입**(위치 무관, 항상 적용). 그 위에 호출부가 ⓐ인라인 / ⓑ테마클래스 / ⓒ완전커스텀으로 덮어쓴다(§2.4, §2.6). **ⓑ가 작동하려면 문서 마운트(C)가 필수** → C와 D가 맞물린다.

### E. 사소하지만 기록할 것
- **무동작 `IDisposable`**: `Noop` 클래스 대신 `Subscription(null)` 공유(`_empty`). 클래스 하나 제거 + 애널라이저 규칙("non-private static readonly는 UPPER_CASE") 회피, `_underscore` 관례 일치.
- **`ShowAt`은 `IPanel` 대신 `VisualElement context`**를 받는다 — anchor가 없어도 문서 루트를 판별해야 하므로.

### F. 마우스 추종도 1급 지원 (`AttachFollow`)
- 처음엔 슬롯 앵커만 `Attach` 한 줄이었고 추종은 호출부가 enter/move/leave를 손으로 달아야 했다 → **API 비대칭(결함).**
- 해결: `AttachFollow(target, provider)` — enter서 콘텐츠 1회 빌드 → move마다 `ShowAt`(커서 추종), leave/detach서 숨김. `Attach`(슬롯)와 대칭.
- 적용: 전투 패널 스킬 = 슬롯 앵커, CharacterDetail 스킬 칩 = 추종(둘 다 공유 빌더 `SkillTooltipContent`).
- **"추종은 깜빡인다"던 처음 우려는 오판**(§2.5): 재마운트는 안 보이는 레이아웃 낭비일 뿐 시각 결함 아님 → 최적화(Mount 가드)는 보류.

### G. 툴팁 스타일 가이드 + 기본 스킨 정식화 (A의 "나중에"를 처리)
- A에선 스타일을 미뤘는데, 이후 **기본 스킨**을 정했다. 단 **리치 툴팁(스킬·건물)은 여전히 호출부 소유**(A 유지) — 정한 건 "기본"뿐.
- `style_guide §6 툴팁`에 **방향·규칙**만 추가(가이드는 변수 전체 목록이 아니라 방향/규칙용 — 변수 단일 출처는 `EoB_Variables.uss`).
- 기본 스킨(`.tooltip` + `__title/__label/__sep/__body/__num/__hint`, `--danger`): 반투명 배경 + 명확한 골드 테두리 + 읽기 쉬운 구분.
- 반투명 위해 신규 변수 **`--bg-overlay`**(`rgba(10,9,7,0.94)`) 1개 — 일반 `--bg-*` 패밀리 **역할 토큰**(툴팁 전용 도메인은 안 만듦; 일반 bg는 전부 불투명이라 alpha 변수가 필요했음).
- "도메인 평탄화"는 보류 — 기존 `--slot-*`/`--echo-*`/`--chain-*`은 진짜 의미색이라 합칠 역할 토큰이 없음(실익 적고 위험만).
- 장비 툴팁을 기본 스킨으로 전환(문자열 오버로드 → 자동 주입), 옛 `equip-slots__tooltip` 클래스 삭제.

## 4. 대안과 안 한 이유

- **전용 오버레이 UIDocument(고 sortingOrder) 1개**: 진짜 전역 최상위를 보장하지만, 패널 간 좌표 변환 + 씬 오브젝트가 필요해 과하다. 미채택(필요해지면 특정 케이스만).
- **스타일시트 복제**(최상위 유지하며 anchor의 시트를 레이어에 복사): 투명을 **땜질**로 푸는 방식. 마운트를 문서 루트로 내리면 근본 해결되어 폐기.
- **기존 것 즉시 전면 교체**: 보류(사용자 지시). 현재 Equip·Skill 통합, Building·trait 보류.
