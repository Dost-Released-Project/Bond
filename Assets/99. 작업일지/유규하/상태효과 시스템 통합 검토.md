# 상태효과(StatusEffect) 시스템 통합 검토

작성일 2026-06-19 · 상태: 검토용(미착수)

## 결론

가능하고, 프로젝트 구조상 자연스러운 통합이다. 다만 프레이밍을 바꿔야 한다 —
**"리액션 효과를 묶는다"가 아니라 "공유 상태효과 서브시스템을 빼낸다"**가 정확하다.
버프·디버프·봉인·불협조 같은 지속 상태가 이미 같은 패턴으로 4벌 복제돼 있고,
코드 곳곳에 "상태이상 시스템 구현 후 연동" 스텁이 *이미* 박혀 있다.

착수한다면 **버프 + 독/출혈/재생을 묶는 실용 코어(Tier A)부터**가 위험이 낮다.
봉인·불협조까지 합치는 완전 통합(Tier B)은 수익 체감이 커서 뒤로 미루는 걸 권장한다.

---

## 1. 지금 상태 — 같은 패턴이 4벌 복제돼 있음

"일정 턴 지속 → 자기 턴마다 tick → 만료 제거" 형태가 전부 독립 구현이다.
각각 `BaseCharacter` 파셜 + 전용 저장 + `ApplyXxx` + `TickXxx`:

| 상태 | 저장 | 적용 | 틱 | 파일 |
|---|---|---|---|---|
| 버프/디버프 | `_activeBuffs : List<ActiveBuff>` | `ApplyBuff` | `TickBuffs` | `02. Scripts/Buffs/BaseCharacter.Buffs.cs` |
| 리액션 봉인 | `_seals : List<ReactionSeal>` | `ApplySeal` | `TickSeals` | `02. Scripts/ReactionSystem/BaseCharacter.Seals.cs` |
| 불협조 | `_distrust : Dictionary<BaseCharacter,int>` | `ApplyDistrust` | `TickDistrust` | `02. Scripts/ReactionSystem/BaseCharacter.Distrust.cs` |
| 턴 스킵 | `_skipTurns : int` | `RequestSkipTurns` | `ConsumeSkipTurn` | `02. Scripts/ReactionSystem/BaseCharacter.TurnState.cs` |

틱은 `TurnManager.cs` 자기 턴 시작에서 **수동 순차 호출**된다:

```csharp
ownerChar.TickBuffs();
ownerChar.TickSeals();
ownerChar.TickDistrust();
ownerChar.ClearRecentAnomaly();
// ConsumeSkipTurn / TryRunSelfTurnAnomaly / ResetReactionCount ...
```

→ 새 상태를 하나 추가할 때마다 (파셜 + 리스트 + Apply/Tick + TurnManager 한 줄)이 반복된다.
이게 "효과가 흩어져 있다"의 실체다. 통합 명분은 충분하다.

리액션 효과 어휘(`ReactionEffect.cs`)도 이미 이들과 1:1 대응한다 —
`BuffReactionEffect` / `SealReactionEffect` / `DistrustReactionEffect` /
`StressReactionEffect` / `SkipTurnReactionEffect`. 각각이 대응 `ApplyXxx`를 호출할 뿐이다.

## 2. 버프/디버프 절반은 이미 완성돼 있음

스탯 모디파이어 엔진이 이미 일반적이다:

- `StatModifier(StatType, Flat/Percent, value)` — `97. Moon/V2/Scripts/StatModifier.cs`
- `StatController.ApplyModifiers(type, base)` — 합산/곱산 적용
- 파이프라인이 `DamageMultiplier`/`DamageReduction`을 **라이브로 읽음**
  — `BattlePipeLineSo.cs:117`(공격 증가), `:221`(피해 감소)

즉 공격력↑/방어력↑ 같은 버프·디버프는 *이미 데이터로 표현 가능*하다.
버프 시스템을 새로 만드는 게 아니라, 이미 있는 걸 상태효과 그릇에 담는 것뿐이다.

## 3. 진짜 새 기능은 "틱마다 부수효과" 하나 (독·출혈·재생)

현재 모든 `TickXxx`는 **지속만 1 감소**시킨다. 틱에 데미지를 주거나 회복시키는 훅이 없다.
독/출혈/재생(DoT/HoT)에 필요한 건 딱 이것:

- 가상 메서드 `OnTick(owner)`에서 `ReduceHP` / `RecoverHp`를 부를 수 있게 (둘 다 `BaseCharacter`에 이미 있음, 사망 처리 포함)
- **스택 규칙** — 버프는 "더 긴 지속으로 갱신"(현재 `ApplyBuff` 동작), 독은 보통 누적. 상태별 `StackMode` 필요

작은 추가다. 기존 틱 구조에 부수효과 한 줄을 더 부를 수 있게 여는 것.

## 4. 결정적 근거 — 코드가 이미 이 시스템을 기다리고 있음

통합은 "있으면 좋은" 게 아니라 **이미 약속된 미구현 시스템**이다. 발견한 스텁:

- `02. Scripts/Skills/Effects/BuffEffect.cs:12`
  — `// TODO: 버프/상태이상 적용 로직 구현` → **스킬도** 소비자다(리액션 전용이 아님)
- `98. LeeJuno` 맵 이벤트 계열:
  - `EventEffectType.StatusEffect` — "버프·디버프·상태이상 (미구현 시스템 연동 예정)"
  - `StatusEffectEventEffectHandler.cs` — "상태이상 시스템 미구현으로 스텁 처리"
  - `EventEffectData._statusEffectId` — "미구현 — TODO: StatusEffectData SO ID"
- `97. Moon/V1/Scripts/BaseCharacter.cs:20`
  — Supporter 역할 트리거 `TRG_DEF_STATUS (상태이상 시)`: 상태 적용 시 발화 예정 트리거가 GDD에 이미 있음

→ 두 가지 함의:
1. 이 시스템은 **ReactionSystem 안에서 태어나면 안 된다.** `02. Scripts/`에 공유 서브시스템으로 두고
   ReactionEffect · SkillEffect · Map이벤트가 모두 디스패치해야 한다.
2. 암시된 **데이터 레이어**(`StatusEffectData` SO)가 있으면 독/출혈을 DBSO로 저작 가능 —
   여러 소비자가 같은 정의를 공유.

## 5. 순진한 통합이 깨지는 지점 (설계 주의)

평평한 "StatModifier 묶음" 모델로 전부 담으려 하면 깨진다:

- **불협조는 관계형이다** — `Dictionary<BaseCharacter,int>`로 *대상 아군별* 지속을 가진다.
  상태효과 *서브클래스가 고유 상태/쿼리를 갖도록* 허용해야지, 모디파이어 컨테이너로는 못 만든다.
- **봉인은 쿼리가 필요하다** — `IsReactionSealed(reaction)`을 `ReactionSystem.Resolve`가 조회.
  통합 시 `GetStatuses<SealStatus>()` 같은 타입 조회 API 필요.
- **턴 스킵은 소비형 카운터** — 지속 틱과 성격이 다름. 굳이 안 넣어도 됨(경계선).
- **연쇄 카운트·돌발(Anomaly) 플래그는 상태효과가 아니라 리액션 장부다** — 절대 합치지 말 것(과잉 통합 위험).
- **소유권 경계** — 코어 `BaseCharacter`+`StatController`는 Moon(`97.`), `TurnManager`는 LeeJuno(`98.`).
  CLAUDE.md상 둘 다 자유 수정 가능이라 차단막은 아니지만, **베이스 타입은 `02. Scripts/`**에 둬야 경계가 깔끔.
- **세이브** — 버프는 `[JsonIgnore]`(전투 런타임 한정). 통합 리스트도 동일하게 두면 **세이브 포맷 변경 0**.
  (맵 이벤트가 전투 밖까지 지속시키려는 상태는 별개의 더 큰 문제 — 지금 범위에서 분리.)

## 6. 리팩토링 규모

### Tier A — 실용 코어 (권장, 저위험, 대략 1일)

- **신규**: 공유 `StatusEffect` 추상
  (`Id` / `RemainingTurns` / `StackMode` / 가상 `OnApply`·`OnTick`·`OnExpire` / `Description`)
  + `BaseCharacter.Status.cs` 파셜
  (단일 리스트 + `ApplyStatus` / `TickStatuses` / `GetStatuses<T>` + `OnStatusesChanged`)
- **구체**: `StatModifierStatus`(기존 `ActiveBuff` 재표현) · `DamageOverTimeStatus`/`RegenStatus`(신규)
- **버프만** 우선 이관. `ApplyBuff`는 얇은 어댑터로 남겨
  `BuffReactionEffect`·파이프라인·UI 무수정 (파이프라인은 `StatController`만 읽으므로 **건드릴 필요 없음**)
- `TurnManager`에 `TickStatuses()` 한 줄 추가
- 규모: 신규 ~4파일 / 수정 ~2-3파일. **이걸로 독·출혈·재생·버프·디버프가 통합 어휘로 완성.**

### Tier B — 완전 통합 (중위험, 크로스팀 변경 큼, 대략 2-3일)

- 봉인 + 불협조(+ 턴 스킵)까지 서브클래스로 이관, 파셜 3-4개 삭제
- `ReactionSystem` 쿼리(`IsReactionSealed`/`IsUncooperativeWith`)·`TurnManager` 틱 전부 재배선
- 저작 DSL(`ReactionDefBuilder`)·카탈로그 생성기 갱신(효과 클래스 형태 바뀌면)
- **수익 체감**: 불협조/봉인은 어차피 고유 상태·쿼리가 필요해 보일러플레이트만 줄지 복잡도는 안 줄음 → **미루기 권장**

### 선택 — 데이터 레이어 (대략 1일)

- `StatusEffectData` SO(DBSO)로 독/출혈을 데이터 저작 → Skill·Map이벤트 소비자까지 연결
- 기존 `StatusEffectId` 참조들과 맞물림. 추상화가 *실제로 본전을 뽑는* 지점

## 7. 권장 진행 순서

1. **Tier A** 먼저 — 파이프라인·세이브 무변경, 위험 낮고, 원하는 독/출혈/버프/디버프를 바로 줌
2. 베이스는 봉인/불협조가 *나중에* 이관 가능하도록 설계만 열어둠 (`GetStatuses<T>` 등)
3. **데이터 레이어**는 Skill/Map이 실제로 저작을 요구할 때 붙임
