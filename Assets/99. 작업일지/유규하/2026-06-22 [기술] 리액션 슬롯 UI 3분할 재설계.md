# 리액션 슬롯 UI 3분할 재설계 — 기술 문서

날짜: 2026-06-22 · 브랜치: Feature/Reaction · 대상: `CharacterDetail`(마을 캐릭터 상세)의 리액션 슬롯

---

## 무엇을 바꿨나

리액션 슬롯을 **"헤더 + 단일 Description 한 줄 + 슬롯 사이로 끼어드는 인라인 풀"** 구조에서
**"헤더 + 대상/조건/행동 3분할 + 떠있는 드롭다운"** 구조로 교체했다. 기준 이미지: `.claude/역할 리액션 예시.png`.

핵심 차이:
- 한 줄 설명(`def.Description`) → **대상 / 조건 / 행동** 3칸 분할.
- 리액션 선택(카탈로그)과 편집칸 풀이 모두 **슬롯 리스트를 밀던 인라인 풀** → **문서 루트에 떠있는 드롭다운 오버레이**.
- 편집칸 상태를 **3색 + 아이콘**으로 표현(고정 / 편집·할당 / 편집·미할당).
- 리액션 미선택 시 3분할을 숨기고 헤더만 노출.

---

## 최종 슬롯 구조 (0~5, 역할 0-1 / 성향 2-5)

```
reaction-slot-N (row)
 ├ __indicator (역할/성향 색 바)
 ├ __slot-main (column)
 │   ├ __header   ─ __header-label  "역할 리액션 - [이름]"   (역할만 클릭→카탈로그)
 │   └ __parts (row, 미선택 시 display:none)
 │       ├ __target-part    : 라벨"대상" / __target / __target-icon(ui:Image)
 │       ├ __condition-part : 라벨"조건" / __condition
 │       └ __action-part    : 라벨"행동" / __action / __action-icon(ui:Image)
 └ __echo  ◇/◆ 다이아몬드 (준비 상태)
```

칸 ↔ 편집슬롯 매핑:
- **대상** = `ObserveTargetEditableSlot` 있으면 편집 가능, 없으면 고정.
- **조건** = 항상 고정(편집 불가, 트리거 서술).
- **행동** = `ActionSkillEditableSlot` 있으면 편집 가능, 없으면 고정.

## 1. 데이터: 3분할 문구 (`ReactionDefinitionSO`)

```csharp
public string TargetTextOverride;     // 비우면 파생
public string ConditionTextOverride;
public string ActionTextOverride;

public (string target, string condition, string action) ResolvePartTexts();
```

- **오버라이드 우선, 없으면 파생**:
  - 조건 = `ConditionTextOverride` ?? `Template.Trigger?.Description`
  - 대상 = `TargetTextOverride` ?? (관찰 편집슬롯 有 ? `"지정 아군"` : ObserveFilter 서술)
  - 행동 = `ActionTextOverride` ?? (행동 편집슬롯 有 ? `"행동 선택"` : `Template.BaseEffect?.Description`)
- 텍스트는 **할당 여부와 무관**하게 동일. 할당이 바꾸는 건 색·아이콘뿐.
- 기존 에셋은 오버라이드 빈 값 → 파생 폴백이라 **하위 호환**(SO 필드 추가라 세이브 영향 없음).

## 2. 상태 표현 (`CharacterDetailPresenter.UpdateEditablePart`)

칸 상태 = `Fixed`(편집 불가) / `Assigned`(편집·할당) / `Unassigned`(편집·미할당).

- **3색** (USS `char-detail__slot-part-val--{fixed|assigned|unassigned}`):
  - 고정 = `--text-secondary`
  - 편집·할당 = `--slot-role-dot` / `--slot-trait-dot`
  - 편집·미할당 = **`--slot-unset`(빨강)** — 리액션 불발 경고
- **아이콘**(`__*-icon`, 문구 아래):
  - 고정 = 숨김
  - 편집·할당 = 실제 아이콘(대상=아군 `EffectiveIdleImageAddress`, 행동=스킬 `IconAddress`) + 네이티브 툴팁(이름)
  - 편집·미할당 = 빈 빨강 테두리 박스(`--slot-part-icon--empty`)
- **다이아몬드 echo**(`SetEcho`): 완성 `◆`(슬롯색) / 미완성 `◇`(빨강) / 빈 슬롯 `◇`(회색).

## 3. 드롭다운 (`SlotDropdown` + Presenter)

- `SlotDropdown`(Plain C# `IDisposable`): 문서 루트에 **절대배치 오버레이**. `Open(anchor, buildItems)` / `Close` / 바깥클릭 닫기(루트 PointerDown capture) / 앵커 아래 배치+flip+패널경계 clamp(`TooltipPopup`과 동일 수학).
- 같은 앵커 재클릭 = 토글(`ToggleDropdown`, `CurrentAnchor` 비교).
- 항목은 커스텀 VE 행(`MakeDropdownItem`: 아이콘 + 이름 + 부제):
  - **카탈로그**(헤더, 역할만): 리액션 이름 + 대상·조건·행동 미리보기 부제, "해제" 항목.
  - **대상**: 아군 초상 + 이름.
  - **행동**: 스킬 아이콘 + 이름 + `TooltipPopup.AttachFollow(…SkillTooltipContent.Build)` 리치 툴팁 재사용.
- 구 `TogglePool`/`CloseAllPools`/`Build*Chips`/`inline-pool-N` 전부 제거.

> 인라인 풀이 아니라 오버레이로 간 이유 / TooltipPopup을 직접 못 쓴 이유는 지식 문서 참고.

## 4. 아이콘 로딩 (Addressables)

- `LoadSlotIcon`: **재로드 가드** — `prev == address`면 생략. 고정 슬롯 아이콘은 요소가 재사용되므로 중복 `LoadAssetAsync`(핸들 ref 누적)·중복 로드 방지.
- `TryLoadSpriteAsync(address)`(공유): **`LoadResourceLocationsAsync`로 키 존재 먼저 확인** → 유효할 때만 `LoadAssetAsync`. 없는 키일 때 Addressables가 자체적으로 찍는 `InvalidKeyException` 로그를 피하고 우리 경고 한 줄만 남긴다.
- 슬롯/드롭다운 두 로더가 이 헬퍼 공유. 경합 가드(`_targetIconAddr`/`_actionIconAddr` 마지막 요청 주소 비교)로 stale 응답 폐기.

> Addressables의 dedup·핸들·churn·존재확인 메커니즘 상세는 지식 문서 참고.

## 남은 작업 (Unity 에디터)

1. `ReactionDefinitionSO` 인스펙터에서 대상/조건/행동 **오버라이드 문구 저작** — 파생 기본값이 디버그체(조건=`"X && Y"`, SkillCast=`"스킬 #N"`)라 이미지처럼 읽히려면 채워야 함(비워도 동작은 함).
2. 아군 초상·스킬 아이콘 Addressable 주소가 실제로 존재하는지 확인(없으면 빈 칸 + 경고 1줄).
