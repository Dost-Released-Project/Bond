# 문제와 해결

## 문제 1. 모든 시스템이 자기 책임으로 Addressables를 직접 호출하던 구조

### 현황

`DataBaseSO`와 그 서브클래스(11개: `SkillDataBaseSO`, `ClassDataBaseSO`, `ConsumableDataBaseSO`, …)는 잘 정의되어 있었지만, **로드 책임이 분산**되어 있었다. 코드베이스를 훑은 결과 같은 패턴이 9곳 이상에서 반복되고 있었다:

```csharp
// InventoryView, ExpeditionInventoryView, InventoryBase.AddItemId, ConstructionUI,
// StageCoach, AdminTestTool, S1Test, MonsterFactory, TownScope, RootScope_KWT ...
var conHandle = Addressables.LoadAssetAsync<ConsumableDataBaseSO>("ConsumableDataBase");
var accHandle = Addressables.LoadAssetAsync<AccessoryDataBaseSO>("AccessoryDataBase");
await Task.WhenAll(conHandle.Task, accHandle.Task);
BaseItem item = accHandle.Result.GetSO<BaseItem>(id) ?? conHandle.Result.GetSO<BaseItem>(id);
```

### 무엇이 문제인가

1. **중복 로드 위험** — 캐싱이 없어 같은 DB를 여러 곳에서 다시 로드할 수 있다. 특히 `InventoryBase.AddItemId`는 호출될 때마다 Consumable/Accessory DB를 다시 로드한다 (사실상 버그성 패턴).
2. **호출자가 Addressable 키 문자열을 알아야 함** — `"ConsumableDataBase"` 같은 문자열이 코드 곳곳에 박힘. 키가 바뀌면 다 찾아 고쳐야 한다.
3. **호출자가 어느 DB에 어떤 SO가 있는지 알아야 함** — `BaseItem` 하나 찾으려고 Consumable과 Accessory DB를 모두 로드하고 두 번 `GetSO`를 시도하는 코드가 곳곳에 있었다.
4. **부트스트랩 / 동기 / 비동기가 뒤섞임** — 어떤 곳은 `WaitForCompletion()`, 어떤 곳은 `await handle.Task`. 의도가 통일되지 않음.

### 해결: 중앙 레지스트리

`DBSORegistry`를 도입해 위 책임을 흡수했다. 핵심 API:

- `GetSO<T>(id)` — ID로 SO 한 줄 조회. 어느 DB에 있든 알아서 찾음.
- `QuerySO<T>(predicate)` — 모든 DB 가로지른 조건 검색.
- `GetDb<T>(key)` / `GetDb<T>()` — DB 인스턴스 자체가 필요한 경우.
- `LoadAsync<T>(key)` / `LoadSync<T>(key)` — 비동기/동기 로드.
- `PreloadAsync(params Type[])` / `PreloadAsync(params string[])` — 부트스트랩 일괄 로드 (동기 동등 `PreloadSync`).
- `PreloadByLabelAsync(label)` / `PreloadByLabelSync(label)` — Addressables 라벨 단위로 일괄 로드.
- `Register(key, db)` — RootScope의 Inspector 참조 SO를 외부에서 주입.

### 문제 1의 4개 항목 중 어디까지 해결되었나 (정직한 평가)

| 항목 | 상태 | 비고 |
|---|---|---|
| 중복 로드 위험 | **해결** | Dictionary 캐시 + Task 공유. 같은 키 두 번 로드 안 됨. |
| 호출자가 Addressable 키 문자열을 알아야 함 | **부분 해결** | 컨벤션 케이스(`GetDb<T>()`, `LoadAsync<T>()`, `PreloadAsync(typeof(...))`)는 문자열 안 씀. 다만 raw `DataBaseSO`나 같은 타입 여러 DB는 본질적으로 string 키가 식별자라 숨길 수 없다. |
| 호출자가 어느 DB에 어떤 SO가 있는지 알아야 함 | **해결** | `GetSO<T>(id)`가 모든 DB 가로질러 탐색. 호출자는 DB 자체를 의식할 필요가 없음. |
| 부트스트랩/동기/비동기가 뒤섞임 | **부분 해결** | `PreloadAsync` / `PreloadSync` 두 진입점으로 명시화. 다만 호출자가 어느 쪽 호출할지는 여전히 컨텍스트(async 가능 여부)에 따라 판단해야 함. |

호출자는 부트스트랩에서 `PreloadAsync(typeof(SkillDataBaseSO), ...)` 한 번 호출하고, 이후엔 `DBSORegistry.GetSO<SkillData>(id)` 한 줄로 끝낸다.

## 문제 2. 초기 설계가 "타입 = DB 식별자"를 암묵적으로 강제

### 발견

초기 구현은 내부 저장소를 `Dictionary<Type, DataBaseSO>`로 두고 `typeof(T)`를 키로 썼다. 한 줄 요약: **"하나의 서브클래스 = 하나의 DB 인스턴스"** 가정.

```csharp
// 초기 버전
private static readonly Dictionary<Type, DataBaseSO> _databases = new();

public static T GetDb<T>() where T : DataBaseSO
    => _databases.TryGetValue(typeof(T), out var db) ? db as T : null;
```

### 어떤 경우에 깨지나

리뷰 중 다음 두 케이스에서 이 가정이 무너진다는 점이 드러났다:

1. **raw `DataBaseSO`로 만든 에셋 여러 개**
   `DataBaseSO`에 이미 `[CreateAssetMenu]`가 붙어 있다. 누구든 베이스 타입 그대로 SO를 만들 수 있고, 그 경우 모두 `typeof(DataBaseSO)`를 키로 갖게 되어 서로를 덮어쓴다.

2. **같은 서브클래스 인스턴스 여러 개**
   예: `ClassDataBaseSO` 두 개 — 영웅용/적용으로 분리하는 디자인. 또는 챕터별로 다른 `EventDataBaseSO`. Type만으로는 둘을 구분할 수 없다.

### 해결: 키를 Addressable string으로

올바른 식별자는 **Addressable 키**다. Type은 어디까지나 "관례적 단축형"으로만 인정.

```csharp
// 변경 후
private static readonly Dictionary<string, DataBaseSO> _databases = new();
```

그 위에 두 종류의 API를 쌓았다:

- **명시적 키 API (일반)**
  ```csharp
  var heroes   = DBSORegistry.GetDb<ClassDataBaseSO>("HeroClassDB");
  var villains = DBSORegistry.GetDb<ClassDataBaseSO>("VillainClassDB");
  ```

- **컨벤션 단축형 (T → 키 자동 매핑)**
  ```csharp
  // ResolveKey<T>(): TypeName에서 "SO" 접미사 제거
  // SkillDataBaseSO → "SkillDataBase"
  var skillDb = DBSORegistry.GetDb<SkillDataBaseSO>();
  ```

부수 효과: `PreloadAsync`도 string-keyed가 되면서 리플렉션이 제거됐다.
`Addressables.LoadAssetAsync<DataBaseSO>(key)`로 베이스 타입으로 로드해도 반환되는 인스턴스는 실제로는 서브클래스(예: `SkillDataBaseSO`)다. C# 런타임 타입이 보존되니 후속 `as SkillDataBaseSO`가 그대로 통한다. 그래서 PreloadAsync 시점에는 타입을 몰라도 된다.

## 문제 3. `PreloadAsync(...).Wait()` 호출 시 Unity가 "응답 없음"

### 증상

테스트 도중 다음과 같은 코드로 호출하니 에디터가 멈췄다:

```csharp
public void Start()
{
    DBSORegistry.PreloadAsync("ClassDataBase", "SkillDataBase").Wait();
    // 도달 못함
}
```

### 원인: async-over-sync 데드락

Unity의 기본 `SynchronizationContext`는 `UnitySynchronizationContext`다. 이것은 `await` 이후의 continuation(나머지 코드)을 **메인 스레드의 다음 프레임 큐**로 post한다.

흐름은 이렇다:

1. 메인 스레드가 `PreloadAsync(...).Wait()` 호출 → 메인 스레드가 Task 완료를 기다리며 **블록**된다.
2. Task 내부에선 `await Task.WhenAll(...)` 진행 중. 완료되면 continuation을 메인 스레드의 SyncContext에 post한다.
3. continuation을 실행할 메인 스레드는 (1) 때문에 잠겨 있어서 큐를 처리하지 못한다.
4. Task는 자신의 continuation이 실행되어야 완료 상태로 마킹되는데, 그 continuation이 실행 안 되니 영원히 미완료.
5. → `.Wait()`은 영원히 반환되지 않음 → 데드락 → "응답 없음".

여기에 Addressables가 가세하면 더 명확해진다 — Addressables의 로드 진행 자체가 메인 스레드 frame loop에 의존하는 경우가 있어서, 메인 스레드가 막혀 있으면 로드 자체가 흘러가지 않는다.

### 해결책 둘

**A) async 경로로 옮긴다 (권장)**

호출자를 `async`로 바꾸고 `await`로 받는다. await은 메인 스레드를 release하므로 continuation을 받을 수 있다.

```csharp
private async void Start()
{
    await DBSORegistry.PreloadAsync("ClassDataBase", "SkillDataBase");
    payload.Clear();
    FillRosterFromPreset();
}
```

**B) 동기 전용 메서드 `PreloadSync` 추가**

Addressables는 메인 스레드에서 안전하게 동기 대기할 수 있도록 `WaitForCompletion()`이라는 전용 동기 API를 제공한다. 이건 `Task.Wait()`와 달리 SyncContext에 갇히지 않고 **내부적으로 펌프를 돌려 로드 작업을 메인 스레드에서 진행시킨다.**

이 메서드를 활용해 `PreloadSync`를 추가했다:

```csharp
public static void PreloadSync(params string[] keys)
{
    var inFlight = new List<(string key, AsyncOperationHandle<DataBaseSO> handle)>(keys.Length);
    foreach (var key in keys)
    {
        if (string.IsNullOrEmpty(key) || _databases.ContainsKey(key)) continue;
        inFlight.Add((key, Addressables.LoadAssetAsync<DataBaseSO>(key))); // 시작만
    }
    foreach (var (key, handle) in inFlight)
    {
        var asset = handle.WaitForCompletion();  // 차례로 완료 대기
        Store(key, asset, handle);
    }
}
```

핸들을 먼저 다 만들어 두면 Addressables가 각 로드를 동시에 시작하므로 가능한 한 병렬화된다. `WaitForCompletion()`은 메인 스레드 펌프와 안전하게 협력한다.

### 정리: `Task.Wait()` vs `WaitForCompletion()`

| 항목 | `Task.Wait()` | Addressables `WaitForCompletion()` |
|---|---|---|
| 메인 스레드 블록 | O | O |
| continuation post 필요 | O — 메인으로 post되는데 메인이 막혀 있어 데드락 | X — 내부 펌프로 작업을 메인에서 직접 진행 |
| Unity 메인 스레드에서 안전 | 위험 | 안전 |

**결론**: Unity에서 "동기 흉내"가 필요할 때는 절대 `Task.Wait()` / `Task.Result`를 쓰지 말 것. Unity가 제공하는 전용 동기 API를 사용한다.

이 점이 워낙 자주 실수가 나는 지점이라 `PreloadAsync`의 XML 주석에도 명시해 두었다:

```csharp
/// 주의: 절대 .Wait() / .Result로 받지 말 것. 메인 스레드 데드락이 발생한다.
/// 동기 경로가 필요하면 PreloadSync를 사용한다.
```

# 부차 논의 (오늘 다룬 개념들)

문제 해결 과정에서 나눈 학습성 토픽 정리.

## 1. UniTask로 바꿀까?

코드베이스 일부에 UniTask가 이미 도입되어 있어 고민했지만, **이 클래스에는 의미 없다**고 판단했다.

이유:
- `LoadAsync`가 await되는 빈도가 매우 낮다 (부트스트랩 1회 + DB별 첫 로드 1회씩, 총 10~20회 수준).
- 정상 운용 흐름은 동기 `GetDb<T>()` / `GetSO<T>(id)`. 여긴 Task 자체가 안 생긴다.
- UniTask는 매 프레임/매 틱 await하는 경로 — `BattleManager`, `TurnManager`, `StageLoader` 등 — 에서 진가가 나온다. 데이터 로더는 그 카테고리가 아니다.
- Addressables의 `handle.Task`가 이미 `Task`라서 UniTask로 가도 한 번 변환을 거쳐야 한다 — 이득 미미.
- 호출처 9곳이 모두 `Task` 기반이라 시그니처가 어긋나면 변환 코드가 흩어진다.

## 2. async/await의 할당이란?

UniTask 논의 중 "Task 할당이 무엇이냐"는 질문을 받고 정리한 내용.

컴파일러가 `async Task<T>` 메서드를 만들 때 발생하는 할당:

1. **`Task<T>` 인스턴스 자체** — 클래스(참조 타입)라 매 호출마다 힙 할당.
2. **상태 기계 박싱** — 컴파일러가 async 메서드를 struct 상태 기계로 변환한다. await이 실제로 일시정지하면 그 struct가 힙으로 박싱된다 (재개를 위해 살아 있어야 하므로).
3. **continuation 델리게이트** — await이 멈출 때 `Action` 콜백 생성 및 awaiter에 등록.
4. **SyncContext 캡처/복원** — 컨텍스트 관련 부속 할당.

`UniTask`는 struct 기반 + 상태기계 풀링 + SyncContext 미캡처로 이 모두를 제거하거나 줄인다. 다만 자주 await되지 않는 경로에서는 효과 없음.

## 3. `await` vs `.Wait()`의 본질적 차이

| 항목 | `await` | `.Wait()` |
|---|---|---|
| 스레드 점유 | release (비차단) | hold (차단) |
| 메인 스레드 게임 루프 진행 | 계속 진행 | 멈춤 |
| continuation | SyncContext에 등록, 나중에 실행 | 그냥 블록 |
| 예외 전파 | 원래 타입 그대로 throw | `AggregateException`으로 래핑 |
| Unity 메인 스레드에서 안전성 | 안전 (잘 쓰면) | **위험 — 데드락** |
| 사용 조건 | `async` 키워드 필요 | 어디서나 호출 가능 |

한 줄 요약:
- `await` = "끝나면 알려줘, 난 다른 일 할게"
- `.Wait()` = "끝날 때까지 여기서 기다린다" — Unity 메인 스레드에선 거의 항상 잘못된 선택

## 4. 키 표현 방식 — 문자열 리터럴 / `const` / `typeof` / 라벨

문제 2를 해결하며 키를 string으로 잡고 나니 자연스럽게 "그 string을 어디서/어떻게 표기할 것인가"가 다음 주제가 됐다. 네 가지 방식을 비교했다.

### (a) 문자열 리터럴 — 출발점

```csharp
Addressables.LoadAssetAsync<SkillDataBaseSO>("SkillDataBase");
```

문제: 9곳에 흩어져 있어 키 변경 시 일괄 수정이 필요하고, 오타가 런타임까지 침묵한다. 이 상태가 마이그레이션 전 코드베이스의 모습.

### (b) `const` 문자열 클래스

```csharp
public static class DBSOKeys
{
    public const string Skill      = "SkillDataBase";
    public const string Class      = "ClassDataBase";
    // ...
}
DBSORegistry.PreloadAsync(DBSOKeys.Skill, DBSOKeys.Class);
```

장점:
- IDE 자동완성과 컴파일 안전성
- 오타가 런타임이 아닌 컴파일 단계에서 잡힘
- raw `DataBaseSO`나 동일 타입 다수처럼 **타입으로 식별 불가능한 케이스**에도 적용 가능

단점:
- 클래스 추가 시 const도 같이 추가해야 — 이중 관리
- 클래스명과 키 문자열의 일관성을 사람이 지켜야 함 (오기입 가능)
- Rename refactor가 따라오지 않음

### (c) `typeof()` + 컨벤션 매핑

```csharp
DBSORegistry.PreloadAsync(typeof(SkillDataBaseSO), typeof(ClassDataBaseSO));
```

내부적으로 `ResolveKey(Type)`이 `TypeName - "SO"` 변환을 수행해 string 키를 얻는다.

장점:
- 새 클래스 추가 시 자동 — 별도 등록 없음
- IDE Rename refactor가 따라옴
- 클래스명과 키가 항상 동기화 (컨벤션이 강제)

단점:
- **raw DataBaseSO나 동일 타입 다수 케이스에선 동작 불가** — 타입만으로 식별 불가능
- "TypeName-SO" 컨벤션을 모든 DBSO가 지킨다는 가정에 의존

### (d) Addressables 라벨

```csharp
await DBSORegistry.PreloadByLabelAsync("DBSO");
```

내부적으로 `LoadResourceLocationsAsync(label, typeof(DataBaseSO))`로 라벨이 붙은 모든 위치를 찾아 각각의 `PrimaryKey`로 일괄 로드.

장점:
- **새 DBSO 추가 시 코드 수정 0회.** 에셋에 라벨만 붙이면 다음 부트스트랩에 자동 포함.
- 호출자는 키 목록도, 타입 목록도 알 필요 없음 — 라벨 하나만 알면 됨.
- raw / 동일 타입 다수 케이스도 자연스럽게 포섭됨 (라벨이 식별자가 아니라 분류자라서)
- 추가/삭제가 에디터에서 발생 — 데이터 작업과 결이 맞음

단점:
- 에디터에서 라벨을 붙이는 수작업이 필요. 잊으면 누락되고, 누락은 코드에서 안 보임.
- 어떤 DB가 로드되는지 코드에 명시되지 않아 추적이 약함 — Inspector를 봐야 알 수 있음.
- "DBSO"라는 라벨 자체는 여전히 magic string. 다만 **한 곳에 한 번만** 등장한다는 점이 핵심 차이.

### 결론 — 셋 다 살리되 우선순위 분명히

| 케이스 | 권장 방식 |
|---|---|
| 일반 부트스트랩 (DBSO 카테고리 전체 로드) | (d) 라벨 — `PreloadByLabelAsync("DBSO")` |
| 특정 DB 몇 개만 따로 로드 (테스트, 부분 씬) | (c) typeof — `PreloadAsync(typeof(SkillDataBaseSO))` |
| 동일 타입 다수 / raw DataBaseSO | (b) const — 키를 코드로 관리 |
| (a) 리터럴 | 사용 안 함 — 단발성 디버그 외엔 금지 |

`DBSORegistry`는 (a)를 제외한 세 진입점을 모두 제공한다. 일반 케이스는 라벨로 가고, 안 들어맞는 경우만 다른 진입점으로 빠진다.

### "DBSO" 라벨 magic string은 어디 두나

라벨 이름("DBSO")이 호출 코드에 직접 등장하는 게 거슬리면 const로 한 번 더 감싸도 된다:

```csharp
public static class DBSOLabels
{
    public const string All = "DBSO";
}

await DBSORegistry.PreloadByLabelAsync(DBSOLabels.All);
```

지금은 라벨이 하나뿐이라 과한 추상화로 보여 추가하지 않았다. 라벨이 여러 개로 분기되는 시점에 도입하면 충분.

# 에디터 작업 — Addressables 라벨

`PreloadByLabelAsync("DBSO")`를 쓰려면 에셋 측에 라벨을 붙여 둬야 한다. 한 번만 해두면 새 DBSO 추가 시 코드 수정 없이 부트스트랩에 자동으로 포함된다.

## 절차

1. Unity 메뉴 → `Window > Asset Management > Addressables > Groups`
2. `Data` 그룹 펼치기. 현재 등록된 11개 DBSO 에셋이 보인다:
   - SkillDataBase, ClassDataBase, ConsumableDataBase, AccessoryDataBase,
   - DefaultEquipDataBase, MonsterDataBase, MonsterGroupDataBase,
   - BuildingDataBase, EventDataBase, StatModifierDataBase, JournalDataBase
3. 11개 항목을 Shift 클릭으로 다중 선택
4. Inspector 우측 상단 또는 Groups 창 우측의 **Labels** 영역 클릭 → `+` 버튼 → 새 라벨 이름 `DBSO` 입력 → 생성
5. 다중 선택 상태에서 `DBSO` 라벨 체크박스 활성화 — 11개 항목 모두에 라벨이 붙는다
6. Addressables Groups 창 좌측 상단 **Build** → **New Build** → **Default Build Script** 한 번 실행 (런타임 매니페스트 갱신)

## 호출

부트스트랩 코드(예: `RootScope.Configure`)에서:

```csharp
await DBSORegistry.PreloadByLabelAsync("DBSO");
```

이후엔 어디서나 `DBSORegistry.GetSO<SkillData>(id)` 한 줄.

## 새 DBSO 추가 시 체크리스트

- [ ] Addressables Groups에 에셋 등록되어 있는가
- [ ] `DBSO` 라벨이 체크되어 있는가
- 위 두 가지만 되어 있으면 코드 수정 없이 부트스트랩에 포함된다.

# 미완료 / 후속 작업

`DBSORegistry` 클래스 자체는 완성되었지만, 진짜 효과는 **기존 호출처를 마이그레이션**해야 난다. 마이그레이션 대상과 위험도:

| 위치 | 현재 패턴 | 마이그레이션 난이도 | 비고 |
|---|---|---|---|
| `MonsterFactory.EnsureDb` | `WaitForCompletion` 1회 | 쉬움 | `LoadSync<MonsterDataBaseSO>()` 치환 |
| `StageCoach` | 필드 초기화에서 3개 DB 로드 | 쉬움 | 부트스트랩 후 `GetDb<T>()` |
| `AdminTestTool.Awake` | async + 2개 DB | 쉬움 | `await LoadAsync<T>()` |
| `S1Test.Start` | 동기 1개 DB | 쉬움 | `LoadSync<T>()` |
| `ConstructionUI.Awake` | async 1개 DB | 쉬움 | `await LoadAsync<T>()` |
| `InventoryView.LoadTotalInventory` | async 2개 DB | 중간 | 부트스트랩 보장 후 `GetSO<BaseItem>` |
| `ExpeditionInventoryView.Start` | async 2개 DB | 중간 | 위와 같음 |
| `InventoryBase.AddItemId` | **호출마다** 2개 DB 재로드 (버그성) | 중간 | 캐싱 이득이 가장 큼 |
| `TownScope.Configure` | `WaitForCompletion` 1개 | 중간 | `Register`로 컨테이너 등록 + 레지스트리 동시 등록 |
