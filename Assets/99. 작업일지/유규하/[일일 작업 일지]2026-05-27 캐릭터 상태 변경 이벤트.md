# 문제

`CharacterCombatPanel`(전투 중·탐사 중 상시 표시)과 `CharacterDetail`(우클릭으로 열림) 두 UI 모두에서 캐릭터의 정보가 변해도 갱신이 일어나지 않았다. 구체적으로:

1. **HP/광기 변경** — 전투 중 `ReduceHP / RecoverHp / ReduceInsanity / RecoverInsanity`가 호출돼도 HP/광기 바가 그대로.
2. **스탯 변경** — 장신구 장착 시 칩 visual은 바뀌는데 atk/def/spd 등 숫자는 그대로.
3. **역할/장신구 변경의 우회 경로** — `CharacterDetailController`를 거치지 않고 `BaseCharacter.SetRole`이나 `Accessories[i] = X`를 직접 호출하면 UI가 모른다.

## 근본 원인

`BaseCharacter`가 자기 상태 변경을 **push**하지 않는다. 모든 mutation이 필드를 직접 수정하기만 한다.

```csharp
public void ReduceHP(int amount)
{
    Stat.current_Hp = Mathf.Max(Stat.current_Hp - amount, 0);
    // ... 이벤트 없음
}
```

기존에 갱신이 일어나는 경로는 세 가지뿐이었다:

| 경로 | 트리거 | 한계 |
|---|---|---|
| `SetCharacter()` 호출 (캐릭터 전체 교체) | `ICharacterSelector.OnSelectionChanged` | 같은 캐릭터의 상태 변경은 못 잡음 |
| `CharacterDetailController.SetRole / SetReaction*` | 컨트롤러를 거친 변경 | 우회 경로는 미반영 |
| `CharacterItemService.OnEquipmentChanged` | 장비 서비스 거친 장신구 변경 | 칩만 갱신 (스탯 숫자는 미반영) |

세 경로 모두 본질적으로 **외부에서 변경을 알려줘야** UI가 안다. `BaseCharacter` 자체는 침묵.

# 해결

## 1. `BaseCharacter`에 상태 변경 이벤트 5종

```csharp
public event Action<BaseCharacter> OnHpChanged;
public event Action<BaseCharacter> OnInsanityChanged;
public event Action<BaseCharacter> OnStatRecalculated;
public event Action<BaseCharacter> OnRoleChanged;
public event Action<BaseCharacter> OnAccessoriesChanged;
```

그리고 mutation 메서드 끝에서 invoke:

```csharp
public void ReduceHP(int amount)
{
    if (IsDead) return;
    Stat.current_Hp = Mathf.Max(Stat.current_Hp - amount, 0);
    OnHpChanged?.Invoke(this);
    // ...
}

public void SetRole(RoleType role) { /* ... */ OnRoleChanged?.Invoke(this); }
public void CalcStat()             { /* ... */ OnStatRecalculated?.Invoke(this); }
```

### 설계 결정

- **시그니처 `Action<BaseCharacter>`** — 기존 `OnDead`, `onPlayerTurnStarted`와 동일한 컨벤션. 핸들러에서 캐릭터 인자를 그대로 받아 쓸 수 있어 `_character == c` 같은 필터링도 가능.
- **세분화된 이벤트 (단일 `OnChanged` 아님)** — 부분 갱신 가능. HP 바뀔 때 전체 view를 다시 그릴 필요 없음. 비용도 낮음 — 핸들러 추가/제거만.
- **`[JsonIgnore]` 안 붙임** — `[JsonIgnore]`는 Field/Property에만 유효한 attribute. `event` 선언에 붙이면 컴파일 에러. 기존 `onPlayerTurnStarted` 이벤트도 `[JsonIgnore]` 없이 선언돼 있고, Newtonsoft는 이벤트를 기본적으로 직렬화하지 않으므로 무방.

## 2. host presenter가 직접 구독

`CharacterCombatPanelPresenter`와 `CharacterDetailPresenter`는 자신이 소유한 view element(HP 바, 스탯 라벨, identity)에 대응하는 이벤트만 구독한다.

```csharp
private void AttachCharacterEvents(BaseCharacter character)
{
    if (character == null) return;
    character.OnHpChanged        += HandleHpChanged;
    character.OnInsanityChanged  += HandleInsanityChanged;
    character.OnStatRecalculated += HandleStatRecalculated;
    character.OnRoleChanged      += HandleRoleChanged;
}

private void HandleHpChanged(BaseCharacter c)        => RefreshHpBar(c.Stat.current_Hp, c.Stat.max_Hp);
private void HandleInsanityChanged(BaseCharacter c)  => RefreshInsanityBar(c.Insanity, 100);
private void HandleStatRecalculated(BaseCharacter c) { RefreshStats(); RefreshHpBar(...); }
private void HandleRoleChanged(BaseCharacter c)      => RefreshIdentity();
```

캐릭터 교체 시점(`BindCharacter` / `SetCharacterInternal`)에 이전 캐릭터 detach → 신규 attach. `OnDestroy`에서도 detach.

### 설계 결정

- **단일 `_character` 필드만 유지** — 처음엔 `_character`와 `_subscribedCharacter`를 분리할까 고민했지만, `Detach → 재할당 → Attach` 순서로만 변경하면 둘은 항상 동기화되므로 분리 의미 없음. 단순한 게 정답.
- **부분 갱신 메서드 분리** — `BindCharacter` 안에 있던 통짜 갱신 로직을 `RefreshIdentity` / `RefreshStats`로 분리. 이벤트 핸들러가 필요한 부분만 호출.

## 4. 장신구 변경 채널을 `BaseCharacter.OnAccessoriesChanged`로 통합

기존엔 `CharacterItemService.OnEquipmentChanged` (글로벌 서비스 이벤트) → `CharacterDetailController.OnAccessoryChanged` (forwarder) → presenter 구독. 두 단계 forwarder를 거치고, 글로벌 이벤트라 "내 캐릭터의 변경인가?" 필터링이 암묵적으로 필요했다 (운 좋게 `_character`로 무조건 갱신해서 동작).

`BaseCharacter.OnAccessoriesChanged` 도입으로 정리:

- `CharacterItemService`의 6군데 `target.Accessories[i] = X` 직접 대입을 `target.SetAccessory(i, X)` 메서드 경유로 변경
- `CharacterItemService.OnEquipmentChanged` 이벤트 제거 — 구독자가 모두 사라짐 (V2 `EquipmentSlotUI` 삭제 + 컨트롤러 구독 제거)
- `CharacterDetailController.OnAccessoryChanged` 이벤트 제거 — forwarder 역할이 사라짐

이득:
- 캐릭터별 자동 필터링 (글로벌 → per-instance)
- 우회 변경 캐치 (`SetAccessory`만 통과시키면 됨)
- 두 단계 forwarder 제거

## 5. `EquipSlotsPresenter`가 `OnAccessoriesChanged` 자체 구독 (IDisposable)

처음엔 host presenter가 `OnAccessoriesChanged`를 구독해서 `_equipSlots.SetCharacter(c)`로 forwarder 한 번 더 호출하게 만들었다. 그런데 두 host의 `HandleAccessoriesChanged`가 똑같이 위임만 하는 boilerplate였다.

**소유 원칙**에 따라 정리: HP 바·스탯 라벨을 소유한 host가 HP/스탯 이벤트를 구독하는 것과 대칭으로, 부속품 칩 element를 소유한 `EquipSlotsPresenter`가 `OnAccessoriesChanged`를 직접 구독해야 한다.

```csharp
public class EquipSlotsPresenter : IDisposable
{
    public void SetCharacter(BaseCharacter character)
    {
        DetachCharacterEvents(_character);
        _character = character;
        AttachCharacterEvents(_character);
        RefreshAll();
    }

    public void Dispose()
    {
        DetachCharacterEvents(_character);
        _character = null;
    }

    private void HandleAccessoriesChanged(BaseCharacter c) => RefreshAccessories();
}
```

host는 `OnDestroy`에서 `_equipSlots?.Dispose()`만 호출하면 됨. `HandleAccessoriesChanged` 핸들러 자체가 호스트에서 사라짐.

### 왜 `IDisposable`인가

`EquipSlotsPresenter`는 plain C# 클래스라서 Unity의 OnDestroy를 못 받는다. 장수명 캐릭터(파티 멤버 등)가 죽은 presenter의 핸들러 델리게이트를 계속 붙들고 있으면 메모리 누수. host가 명시적으로 정리할 진입점이 필요하다 — `IDisposable`이 가장 관례적이고 의도가 명확.

대안 (`SetCharacter(null)`로 detach만 트리거)도 가능하지만 의도가 흐릿해서 채택 안 함.

# 사용자 후속 커밋 — `SetAccessory`를 단일 트랜잭션으로

내가 도입한 `SetAccessory`는 단순히 배열 대입 + 이벤트 발사만 했다:

```csharp
public void SetAccessory(int index, AccessoryItem item)
{
    if (index < 0 || index >= Accessories.Length) return;
    Accessories[index] = item;
    OnAccessoriesChanged?.Invoke(this);
}
```

호출자(`CharacterItemService`)가 여전히 `acc.OnUnequip(target)` / `accItem.OnEquip(target)` / `UpdateHeroStats(target)`를 수동으로 순서 맞춰 호출해야 했다.

사용자가 이 부분을 `SetAccessory` 안으로 흡수:

```csharp
public void SetAccessory(int index, AccessoryItem item)
{
    if (index < 0 || index >= Accessories.Length) return;
    Accessories[index]?.OnUnequip(this);   // 이전 아이템 효과 해제
    Accessories[index] = item;
    Accessories[index]?.OnEquip(this);     // 신규 아이템 효과 적용
    CalcStat();                            // 스탯 재계산 (→ OnStatRecalculated 발사)
    OnAccessoriesChanged?.Invoke(this);
}
```

이제 `SetAccessory`가 진정한 **단일 트랜잭션**이다. 호출자는 그냥 `target.SetAccessory(idx, item)` 한 줄. `CharacterItemService`의 모든 호출처에서 `OnEquip / OnUnequip / UpdateHeroStats` 수동 호출이 사라졌다 (6군데 청소).

### 왜 좋은가

- **호출자가 순서를 외울 필요 없음** — 이전엔 "OnUnequip 먼저 → 배열 대입 → OnEquip → CalcStat" 순서를 모든 호출자가 외워야 했고, 한 군데라도 빼먹으면 잠재적 버그.
- **누락 불가능** — 우회 호출자가 새로 생겨도 `SetAccessory` 한 군데만 통과하면 모든 책임이 충족됨.
- **`UpdateHeroStats` 헬퍼가 사실상 무력화** — 이제 단순히 `CalcStat()` 한 줄만 호출. 한 단계 더 가면 헬퍼 자체를 제거하고 호출처에서 직접 `hero.CalcStat()` 호출해도 됨 (별도 정리 안건).

### 부수 효과 — `OnStatRecalculated`가 자동으로 따라옴

`CalcStat()` 내부에서 `OnStatRecalculated`를 발사하므로, `SetAccessory` 한 번 호출하면 `OnAccessoriesChanged`와 `OnStatRecalculated`가 둘 다 발사된다. 둘 다 다른 subscriber가 듣고 다른 view를 갱신:
- `OnAccessoriesChanged` → `EquipSlotsPresenter`가 칩 갱신
- `OnStatRecalculated` → host presenter가 스탯 숫자 갱신

이벤트가 자연스럽게 분기되어 각자 관심사대로 동작한다.

# 최종 데이터 흐름

```
[mutation 발생]
    BaseCharacter.SetAccessory(idx, item)
        ├─ Accessories[idx]?.OnUnequip(this)
        ├─ Accessories[idx] = item
        ├─ Accessories[idx]?.OnEquip(this)
        ├─ CalcStat()
        │     └─ OnStatRecalculated?.Invoke(this) ──→ host presenter: 스탯 라벨 갱신
        └─ OnAccessoriesChanged?.Invoke(this)     ──→ EquipSlotsPresenter: 부속품 칩 갱신

    BaseCharacter.ReduceHP(amount)
        ├─ Stat.current_Hp -= amount
        └─ OnHpChanged?.Invoke(this)              ──→ host presenter: HP 바 갱신

    BaseCharacter.SetRole(role)
        ├─ RoleType = role; battleType = ...
        └─ OnRoleChanged?.Invoke(this)            ──→ host presenter: identity 갱신
```

각 view가 자기가 소유한 element에 해당하는 이벤트만 구독. 호스트 forwarder, 글로벌 서비스 이벤트, 컨트롤러 forwarder 모두 제거됨.
