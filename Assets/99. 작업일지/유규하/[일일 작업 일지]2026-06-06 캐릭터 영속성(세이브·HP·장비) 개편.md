# 2026-06-06 캐릭터 영속성 구조 개편 (세이브 / HP / 장비 / 리액션)

> 양식 무시하고 오늘 작업 + 대화 흐름 그대로 정리.

## 한 줄 요약

캐릭터 저장이 `Roster.Hire/Fire` 에서만 일어나던 걸, **장비 착탈·리액션 편집·HP·광기까지 보존**되도록 구조를 갈아엎음. 핵심은 "데이터 모델은 거의 그대로, **언제/누가 저장을 트리거하는가**를 재설계"한 것. 곁가지로 로드 시 악세서리 효과 누락 버그, HP 비율 왜곡 문제까지 같이 잡음.

---

## 출발점 (문제)

- `Roster : ISaveable<List<BaseCharacter>>` 가 `roster.json` 한 파일로 전체 캐릭터 리스트를 직렬화함.
- `BaseCharacter` 가 `[Serializable]` 라 Weapon/Armor/Accessories/Skills/TraitIds/RoleReactions/TraitReactions/RoleType/Insanity 를 다 들고 있음 → **장비·리액션은 이미 세이브 블롭 안에 들어 있었음.**
- 문제는 **저장 호출이 `Hire`/`Fire` 두 곳뿐**. 장신구 장착, 리액션 편집, 역할 변경 등은 메모리만 바뀌고 디스크에 안 써짐.

→ 즉 "새 저장 포맷이 필요한 게 아니라 `Save(roster)` 를 적시에 더 부르면 되는" **더티/플러시 정책 문제**였음.

## 사전 조사로 확인한 사실

- **편집되는 인스턴스 = Roster 가 들고 있는 인스턴스.** `CharacterItemService` 는 `_characterSelector.Selected`, `CharacterDetailController` 는 선택된 `_character` 를 건드리는데 둘 다 Roster 리스트 원소를 가리키는 참조. → `Save(roster)` 한 방이면 다 잡힘.
- **전투도 같은 인스턴스를 두들김.** `ExpeditionPayload`(RootScope Singleton, `RootScope.cs:19`)의 `Party` 가 `PartyController.currentParty` 의 `List<BaseCharacter>` 를 그대로 참조하고, `BattleStageEntry` 가 `payload.Party[i]` 를 전투에 투입. HP 는 그 인스턴스 위에서 깎임.
- **리액션 편집 단일 chokepoint = `CharacterDetailController`.** Presenter 는 전부 컨트롤러에 위임(읽기만 직접).
- **부트스트랩 순서**: Title 씬에서 `DBSORegistry.PreloadByLabelAsync("DBSO")`(`TitleSceneController:17`) → Town. 로드(역직렬화)는 `BaseSORefConverter` 가 DBSORegistry 로 SO 참조를 해석하므로 **DBSO preload 가 선행해야 함.**
- `Insanity` 는 이미 직렬화 필드(미JsonIgnore)라 데이터는 보존됨 — 누락된 건 `current_Hp` 하나뿐.

## 안 한 접근 (피한 것)

1. **변이 지점마다 `SaveLoadSystem.Save` 흩뿌리기** — 지금 Hire/Fire 누락과 같은 버그 양산. 팀 경계(02.Scripts / 97.Moon)를 넘어 흩어져 새 경로마다 또 빠짐.
2. **매 변이마다 즉시 저장** — `SaveLoadSystem.Save` 가 에디터에서 `AssetDatabase.Refresh()` 까지 부름. 드래그 한 번에 전체 직렬화 + 리임포트. `SetAccessory` 하나가 이벤트 2개 쏴서 중복 저장도 남.

## 확정한 구조

**원칙: 캐릭터 인스턴스 정체성을 게임 전역에서 하나로 유지하고, 변경 성격(메타 vs 세션)에 따라 저장 트리거를 다르게 건다.**

| 상태 | 성격 | 저장 트리거 |
|---|---|---|
| 멤버십(Hire/Fire) | — | 즉시 |
| 장비·역할·리액션 | **메타** (마을 편집) | 캐릭터 이벤트 구독 → 더티 → 디바운스 500ms flush |
| current_Hp·Insanity | **세션** (전투 변동) | 변경 시 저장 안 함. 원정 귀환 `SaveNow` + 종료 시 flush |

### Phase 0 — 수명 이전 (HP 때문에 강제됨)

- `Roster` 를 `TownScope`(Scoped) → **RootScope Singleton** 으로 승격. (`RootScope_HA.cs` 신설 + `RootScope.Configure` 에서 `ConfigureRoster(builder)` 호출)
- 이유: 장비/리액션은 마을 안에서만 편집되지만 **HP 는 마을 밖(전투)에서 바뀜.** 기존 Town-Scoped Roster 는 마을 떠나면 폐기되고 귀환 시 디스크에서 새로 로드 → 전투 결과 유실 + 인스턴스 이중화. Root Singleton 으로 올리면 마을→전투→귀환 내내 동일 인스턴스 유지.
- 지연 생성(첫 resolve = Town)이라 생성자의 로드/기본생성이 Title 의 DBSO preload 이후에 돈다. 소비자(TownUIController/EmbarkPresenter/SupplyManager/S1Test)는 전부 Town 계층이라 부모 스코프 해소로 정상 주입.

### Phase 1 — 변이 chokepoint (`BaseCharacter`)

- 이벤트 2종 추가: `OnEquipmentChanged`(무기/방어구), `OnReactionsChanged`(리액션).
- `SetWeapon()/SetArmor()` 추가 (런타임 교체 UI 생기면 이 한 곳만 통과).
- `SetRoleReaction()/ClearRoleReaction()/RaiseReactionsChanged()` 추가.
- `CharacterDetailController` 의 리액션 변이 5곳(SetRole 프루닝, SelectRoleReaction, ClearRoleReaction, SetObserveTarget, SetActionSkill)을 이 메서드들로 라우팅 → 편집이 영속 트리거를 발화. UI용 `OnReactionChanged`는 유지.

### Phase 2 — 자동저장 권위자 (`Roster`)

- 메타 이벤트(`OnAccessoriesChanged/OnEquipmentChanged/OnRoleChanged/OnReactionsChanged`)만 구독 → `_dirty` → `UniTask.Delay(500ms)` 디바운스 flush. **HP/Insanity 는 구독 안 함.**
- `IDisposable` — 앱/PlayMode 종료 시 미저장분 강제 flush(VContainer 가 RootScope 해제 시 Dispose 호출).
- `SaveNow()` 추가.
- 캐릭터 add/remove 시 Subscribe/Unsubscribe 관리.

### Phase 3 — HP/Insanity 영속 (`BaseCharacter`)

- `current_Hp` 직렬화 미러: `[JsonProperty("CurrentHp")]` private 프로퍼티(get=런타임값, set=`_loadedHp` 보관). Stat 은 `[JsonIgnore]` 라 HP 만 따로 굽고, max_Hp 는 CalcStat 재계산이라 미저장.
- `Init()` 을 **순수 재수화**로: `CalcStat()` → `_loadedHp` 있으면 클램프 복원, 없으면(신규) 풀피. **풀힐(SetHpFull)을 역직렬화에서 분리** — 풀힐은 게임 규칙(마을 휴식/치료)이지 로드 행위가 아님.
- 원정 귀환(`ExpeditionResultService.ProcessExpeditionReturn` → `Roster.SaveNow()`)에서 파티 HP/광기 일괄 flush. (`EmbarkController:114` / `InventoryView:57` 가 호출)

## 대화 중 튀어나온 것들 (시간순)

### 1. "디바운스가 뭐야"
변경 폭주를 즉시 저장으로 직결하지 않고 잠잠해진 뒤 1번으로 묶는 것. 이득은 디스크/`AssetDatabase.Refresh` 폭주 방지. **대가: 미룬 사이 종료되면 마지막 변경 유실 → 그래서 Dispose 강제 flush 와 항상 세트.**

### 2. "IDisposable 의미"
내가 부르는 게 아니라 **VContainer 가 스코프(RootScope=ProjectContext) 해제 시 자동 호출.** Root Singleton 이라 앱/PlayMode 종료 시점에 불림. 디바운스 갭(=`_dirty`인데 아직 안 써진 500ms)을 종료 직전 `Flush()` 로 메우는 안전망. 씬 전환에선 안 불림(Roster 가 Root 라 생존).

### 3. "HP 저장 어떻게" → 수명 결정을 강제
HP 는 장비/리액션과 달리 **마을 밖(전투)에서 바뀜.** 그래서 "current_Hp 직렬화"만으로는 부족하고, 전투 결과를 받을 주체(=Root Singleton Roster)와 원정종료 저장이 필요. 사용자 결정: **(1) 데미지가 마을 귀환 후에도 남나? → 예. (2) 앱 재시작 후에도? → 예.** (DD류 정석: 마을에서 자원/건물로 치료)

### 4. 악세서리 효과 로드 누락 버그 (사용자 발견)
악세는 `OnEquip` 시 `StatController.AddModifiers()` 로 모디파이어를 **push** 하는데, `StatController` 는 직렬화 안 됨(`[JsonIgnore]`). 로드 시 `OnEquip` 이 안 불려 모디파이어 리스트가 빈 채 CalcStat → 효과 누락. (무기/방어구는 `CalculateStat` 이 `Weapon.bonusSTR` 를 직접 **pull** 이라 무사.)
→ HP 와 같은 "push 방식 파생 런타임 상태 재수화 누락". `Init` 에 `ReapplyAccessoryModifiers()`(장착 악세 `OnEquip` 재호출) 추가. Init 은 Build/Restore 에서 1회만 호출이라 중복 적용 없음.

### 5. HP 비율 왜곡 + 숨은 오버플로 버그 (사용자 발견)
악세로 max_Hp 가 크게 변하는데 current_Hp 는 절대값 유지 → 비율 왜곡. 게다가 **`CalcStat` 이 current_Hp 를 클램프 안 해서, +HP 장비 해제 시 `current > max`(120/100 같은 무효 상태) 가능** — 기존부터 있던 버그였는데 이번 수정으로 비로소 드러남.
사용자 선택: **비율 보존.** 장착/해제 시 HP% 유지(80/100 → 120/150 → 80/100), 오버플로 클램프.

### 6. "rescale 을 CalcStat 에 넣는 게 낫지 않냐" (사용자 지적 — 채택)
처음엔 mutator 3곳(SetAccessory/Weapon/Armor)에서 각각 `RescaleCurrentHp(oldMax)` 호출하게 했는데, 사용자가 CalcStat 한 곳에 넣자고 제안. **맞는 지적이라 채택.**
- 내가 우려한 "로드 경로 깨짐" 은 기우였음: 로드 시 Stat 은 fresh(max=0)라 CalcStat 내부 rescale 이 0 클램프 no-op 으로 돌고, 직후 `_loadedHp` 절대값 복원이 덮어씀. 안 깨짐.
- 진짜 이득: mutator 마다 rescale 부르는 건 "저장 흩뿌리기"와 같은 냄새. max 바꾸는 미래 경로(레벨업/버프/장비강화)가 자동으로 비율 보존됨.
- 최종: `CalcStat` 이 `oldMax` 캡처 → `CalculateStat` → `RescaleCurrentHp(oldMax)`(bool 반환) → 변경됐을 때만 `OnHpChanged`. mutator 는 그냥 `CalcStat()` + 자기 이벤트로 단순화.
- 트레이드오프(감수): CalcStat 이 순수계산 아니라 current_Hp 부수효과를 가짐. 순수-프리뷰 호출자가 없고 `max==oldMax` 가드로 드리프트/불필요 이벤트 없어서 OK. 불변식: **Init 은 CalcStat 다음에 HP 복원**(현재 그러함).

## 남은 것 / 결정 필요

- **버프(`BaseCharacter.Buffs.cs` / `ActiveBuff`)도 같은 push 패턴.** 마을까지 영속시킬 거면 악세처럼 Init 재수화 필요. 현재 미처리(전투 휘발성 가정).
- **Weapon/Armor 필드 잠금 안 함** — 런타임 교체 UI 없어 chokepoint 강제(private-set) 안 함. 무기 교체 UI 만들 때 SetWeapon 경유 + 필드 잠금 권장.
- **사망(IsDead) 영속 안 함** — 0 HP 로 로드되지만 dead 플래그는 복원 안 됨. 사망 규칙 별도 결정.
- **OnApplicationPause flush 미적용** — 현재 종료/귀환 flush 로 충분. 모바일 타깃 시 추가.
- **기존 세이브 호환**: 구 `roster.json` 엔 `CurrentHp` 없어 최초 1회 풀피 로드 후 정상 누적.
- `PartyController` 가 별도 `ISaveable`(hashcode 키)인데 실제 save/load 호출처 없음 — 사실상 vestigial. 런타임엔 Roster 인스턴스 참조라 무해. 추후 정리 후보.

## 테스트 체크리스트 (런타임 미검증 — Unity 실행 환경 아니라 직접 확인 요망)

1. 악세/리액션/역할 편집 → 0.5초 후 `roster.json` 갱신.
2. 부분 체력에서 +HP 악세 장착 → HP% 유지, 해제해도 `current>max` 없음.
3. 악세(+STR/HP) 장착 저장 → 재시작 로드 후 스탯에 악세 보너스 반영(누락 버그 해소 확인).
4. 전투에서 HP 깎고 귀환 → `roster.json` 의 `CurrentHp` 가 깎인 값. 재시작 시 그 HP 로 로드(풀피로 안 돌아옴).
5. 마을→전투→귀환 시 편집 유실 없음(인스턴스 유지).
