# 리액션 시스템 재구조화 — 관찰 필터화, Phase 분리, Effect 추상화

날짜: 2026-05-28
브랜치: Feature/Reaction

---

## 출발점

기존 `Reaction` 은 `SubjectCharacterId` 로 관찰 대상 캐릭터를 **특정 ID로 박아두는** 구조였다.
`reaction.Check(context)` 안에서 `BaseCharacter.Dict[SubjectCharacterId]` 를 조회해
한 명의 subject 를 `Trigger.CheckCondition(subject, context)` 에 넘기는 흐름.

문제:
- "아군 누구든 HP가 낮을 때" 같은 **동적 관찰** 불가
- `ReactionPreset` 에 `ObserveFilter (Self/Ally/Enemy/Specific)` 가 미리 선언돼 있는데
  런타임 `Reaction` 이 이걸 안 쓰고 있음 — 절반쯤 예정돼 있던 작업

추가 동기: "아군이 죽을 공격을 받으려 할 때 대신 맞기" 같은
**원본 행동을 변형하는 리액션** 을 도입하고 싶었음 (`SkillIndex` + `ReactionSkillTarget` 만으론 표현 불가).

## 설계 결정 — 단계별 흐름

### 1) 관찰 필터화 (ObserveFilter)

```
E_ObserveFilter
├─ Self        — 자기 자신만
├─ Ally        — 자신 포함 같은 진영 전체
├─ OtherAlly   — 자신 제외 같은 진영       ← 신규
├─ Enemy       — 반대 진영
└─ Specific    — SubjectCharacterId 로 지목
```

- `Ally` 가 self 포함/제외 두 의미 모두 필요 → **OtherAlly 신설**
- 후보 해석 위치는 `Reaction` 이 아니라 **`ReactionSystem`** (side 캐시 주인이 같이 보유)
- `Reaction` 은 순수 데이터로 단순화, `Match`/`ResolveSubjects` 메서드 제거

### 2) 매치 의미론 — A안 (다발) vs B안 (정책 선택)

여러 후보가 조건 만족 시:
- **A안**: 매치 N명 전부 발화
- B안: `E_MatchSelection { First, LowestHp, PrefersContextTarget, ... }` 로 1명 선택

**A안 채택** — B는 A의 degenerate form이라 나중에 좁히기 쉬움.
기존 트리거(`GetHitTrigger`, `DeathBlowTrigger`)는 `SubjectCondition` 이 끼어 있어
어차피 0/1 매치로 자동 좁혀짐 → 회귀 없음.

### 3) 다발 발화 부작용 — 연출 다중화

처음 구현에선 `ReactionExecution` 을 N개 생성 →
`ExecuteReaction` 이 N번 await → **연출 / 1초 딜레이가 N번 재생됨**.
디자인 의도("리액션 1회 발생, 효과는 N명에 적용") 와 어긋남.

수정:
- `(Agent, Reaction)` 쌍 하나 = `ReactionExecution` **하나**
- `MatchedSubjects` (`IReadOnlyList<BaseCharacter>`) 에 매치 전체 보유
- `ExecuteReaction` 의 연출 래퍼(시작 로그 / Delay / 종료 로그) 는 **1회**,
  스킬 적용만 N회 (Effect 가 처리)

### 4) Pre/Post Apply 분리 + Phase

원래 파이프라인은 `Entry → Eva → Crit → Def → ReactionCall → Apply` 였음.
**ApplyStep 앞뒤로 ReactionCall 하나씩** 두기로:

- `PreApply` : 원본 행동에 영향 주는 리액션 (회피 부스트, 데미지 감쇄, **대신 맞기**)
- `PostApply`: 원본 행동의 후속 리액션 (반격, 처치 시 추가 스킬, KillCondition 류)

```
E_ReactionPhase { None, PreApply, PostApply }
```

- `Reaction.Phase` 기본값 = **`None`** (의식적 선택 강제 — 기존 에셋 일시 비활성, 의도된 부작용)
- `ReactionCall.Phase` 기본값도 `None` — SO 에서 명시적으로 지정
- `ReactionSystem.Resolve(context, phase)` — phase 일치하는 리액션만 평가

### 5) Side 캐시

`CharacterSlot.Clear()` 가 `Occupant.CurrentSlot = null` 로 만들어
**사망 시 진영 정보가 사라짐** → PostApply 에서 죽은 `context.target` 을 Ally/Enemy 로 분류 불가.

해결 후보:
1. `BaseCharacter.BattleSide` 필드 추가
2. 슬롯 Clear 안 함
3. **ReactionSystem 자체 캐시** ← 채택

`Register` 시 `agent.CurrentSlot.side` 캡처, `Unregister` 까지 유지.
`SideOf(c)` 는 캐시 우선 + live 폴백 — 늦은 등록도 대응.

### 6) PostApply 후보 풀의 특수 처리

메인 후보 루프는 죽은 캐릭터(`c.IsDead`)를 걸러낸다 — 시체에 대해
`HpBelowCondition` 같은 게 항상 매치하는 노이즈를 피하려고.

그런데 **PostApply 의 `KillCondition + SubjectCondition(Target)` 패턴**
("적 처치 보너스", "아군 사망 시 복수")이 동작하려면 죽은 `context.target` 이 후보에 있어야 함.

해결: 메인 루프 다음에 **fallback 블록** 으로 `context.caster/target` 을 특수 포함.
`seen` HashSet 으로 살아있는 actor 의 중복 yield 방지.

```csharp
// 메인: 살아있는 participants
foreach (var c in Characters) { ... if (c.IsDead) continue; ... seen.Add(c) ... }

// fallback: PostApply 단계의 dead context actors
if (phase == E_ReactionPhase.PostApply)
{
    TryIncludeActor(context.caster, ...);
    TryIncludeActor(context.target, ...);
}
```

### 7) `IsDead` 가드 제거 (`ReactionCall.Execute`)

원본에 `if (context.target == null || context.target.IsDead) return context;` 가 있었음.
git 추적 결과:
- 2026-05-19 김우태 [병합 커밋] 에서 EvasionStep/CriticalStep/DefenseStep 와 한꺼번에 일괄 추가됨
- 원래는 `isEvaded` 도 포함된 가드였는데 회피 시에도 리액션은 평가해야 한다는 발견으로 `isEvaded` 만 빠짐
  ( `BattlePipeLineSo.ShouldBreak` 에 그 변경 이유 코멘트 남아 있음 )
- `IsDead` 가드는 그 정신을 못 따라가고 남아있던 잔재

다른 파이프라인 step("타겟 죽었으면 이 step 계산이 의미 없음")과 달리
ReactionCall 은 의미 결이 다름 — caster 측 리액션은 타겟 상태 무관하게 발화 가능해야 함.
**제거 → `context.target == null` 만 남김**.

### 8) Effect 추상화 (Pre-Apply modifier 메커니즘)

"대신 맞기" 같은 modifier 류 리액션 대응. 두 가지 접근 비교:

| 옵션 | 설명 | 결론 |
|---|---|---|
| **스킬 방식** | `InterceptSkill` 을 만들고 `UseSkill` 안에서 BattleContext 수정 | ❌ |
| **A1: Effect 추상화** | `ReactionEffect` 추상 + 서브클래스 (`SkillCastReactionEffect` / `InterceptReactionEffect`) | ✅ |

스킬 방식의 문제:
1. 새 BattleContext 에 원본 참조 없음 → `parentContext` 같은 필드 추가 필요
2. **불필요한 파이프라인 전체 실행** — Intercept 가짜 스킬을 위해 Entry/Eva/Crit/Def/Apply 가 다 돔
3. **재귀 발화** — 안쪽 파이프라인의 ReactionCall 이 또 트리거를 평가 → 무한 재귀 위험
4. `ApplyStep` 이 가짜 스킬에 대해 실행됨 — 의미와 안 맞음
5. 스킬은 "효과 발생 단위" 인데 "context 수정 단위" 로 쓰는 건 추상화 위반

Effect 추상화 채택 — 원본 context 를 직접 받아 변형, 새 파이프라인 안 돔, 재귀 없음.

In-game editor 우려에 대해서도:
- SubclassSelector 는 에디터 전용 어트리뷰트일 뿐, `[SerializeReference]` 는 런타임 직렬화 OK
- "스킬 vs 모디파이어" 카테고리 분리가 플레이어 UI 에도 직관적
- 화이트리스트 큐레이션도 깔끔

`Reaction` 의 `SkillIndex` / `ReactionSkillTarget` → `SkillCastReactionEffect` 로 흡수.

### 9) PreApply 위치 이동 → rawValue 불필요

처음엔 PreApply 를 `ApplyStep 직전` 에 뒀음. 그 경우 Intercept 가:
- 타겟을 reactor 로 바꿔도 `context.value` 는 이미 원래 ally 의 def 로 차감된 상태
- → reactor 의 def 로 재계산 필요 → `BattleContext.rawValue` (차감 전 값) 보관 + 수동 재계산

이후 **PreApply 를 EntryStep 직후로 이동** 했더니:
- 타겟만 바꾸면 이후 Eva/Crit/Def 가 새 타겟 기준으로 자연 재계산
- `rawValue` 와 수동 재계산 둘 다 제거 가능

트레이드오프 — 트리거 조건들이 raw 값을 보게 됨:

| 조건 | 기존 (Apply 직전) | 새 위치 (Entry 직후) |
|---|---|---|
| `DamageCondition` | post-defense 값 (정확한 예측) | raw 공격력 (defense 차감 전) |
| `HitCondition` | 회피 판정 반영 | 항상 통과 (isEvaded 초기값 false) |
| `CritCondition` | 크리티컬 판정 반영 | 항상 false (isCritical 초기값 false) |

특히 `DeathBlowTrigger` 가 영향 큼 — "raw 공격력 >= HP" 로 바뀌어
defense 가 큰 상황에서 false positive (안 죽을 공격에도 발동). 안전마진 쪽 기울어짐.

**이 부정확함을 받아들이는 쪽으로 결정** — 게임 디자인상 "혹시 죽을지 모를 일격에 대신맞기"
의 안전 발동이 오히려 자연스러움. `rawValue` 코드 제거.

## "아군이 죽을 공격을 대신 맞기" 리액션 만드는 법

인스펙터 설정:
- **Phase**: `PreApply`
- **ObserveFilter**: `OtherAlly` (자신 제외 아군)
- **Trigger**: `DeathBlowTrigger` 또는 그 조건들을 직접 조합
- **Effect**: `InterceptReactionEffect` (파라미터 없음)

작동 흐름:
1. 적이 아군에게 공격 → 파이프라인 진입
2. EntryStep — `context.value` 산출
3. PreApply ReactionCall — Intercept 매치 평가
   - ObserveFilter=OtherAlly 로 아군 후보 풀 구성
   - `SubjectCondition(Target)` 으로 "context.target 인 아군" 만 통과
   - `DamageCondition(Ratio=1)` 으로 "예측 데미지 >= 그 아군 HP" 검증
   - 통과 시 `InterceptReactionEffect.Apply` 호출
4. Intercept 가 `context.target = reactor` (대신맞는 캐릭터) + `context.interceptedFor = 원래 ally`
5. EvasionStep — reactor 의 AGI 로 회피 재판정
6. CriticalStep — reactor 기준 크리티컬
7. DefenseStep — reactor 의 def 로 차감
8. ApplyStep — reactor 에게 데미지 적용

연출 측에선 `context.interceptedFor != null` 이면 끼어들기 모션 재생 (viz 레이어 작업).

## 설계상 짚어둘 점

### `SubjectCondition` 의 위상
`ObserveFilter` 와 축이 다름:
- `ObserveFilter`: **누구를** 후보 풀로 둘 것인가
- `SubjectCondition`: 그 후보가 **context.caster/target 과 일치하는가**

조합 예:
- `Ally + SubjectCondition(Target)` = 스킬 대상이 된 아군 (= 피격받은 아군)
- `Enemy + SubjectCondition(Target)` = 내가 친 적
- `OtherAlly + KillCondition` = 내가 아닌 아군이 죽었을 때 (PostApply)
- `OtherAlly + SubjectCondition(Target) + DamageCondition(1.0)` = 죽을 공격을 받는 아군 (PreApply)

### 다발 발화의 실제 의미
"매치된 N명" 은 트리거 측 개념. 실제 스킬 적용은 `SkillCastReactionEffect.SkillTarget` 에 따라:
- `Caster` / `Target`: 매치 수 무관하게 스킬 1회
- `Observed`: 매치된 N명 각자에게 1회씩
- 연출 래퍼는 항상 1회

### Effect 의 위치
`Reaction` 은 "언제 발화하나"(Phase + Trigger)와 "누구를 관찰하나"(ObserveFilter)만 들고,
"발화 시 뭘 하나"는 전부 `Effect` 가 담당. 향후 다른 modifier
(예: `DamageHalveEffect`, `EvadeForcedEffect`, `RedirectToRandomAllyEffect`) 도
`ReactionEffect` 하나 더 추가로 끝남.

### PreApply 의 위치 결정
