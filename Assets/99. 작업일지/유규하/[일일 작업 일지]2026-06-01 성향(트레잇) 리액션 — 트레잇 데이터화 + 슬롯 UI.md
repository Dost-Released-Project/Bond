# 성향(트레잇) 리액션 — 트레잇 데이터화 + 슬롯 UI

날짜: 2026-06-02
브랜치: Feature/Reaction
선행: `[일일 작업 일지]2026-05-29 리액션 에디터 — 정의 SO 카탈로그 + 역할 슬롯 UI.md`

---

## 목표

역할 리액션에 이어 **성향(Trait) 기반 리액션** 도입.

- 성향과 리액션은 **1:1 매칭**. 보유한 성향이 **모두 자동으로** 성향 리액션 슬롯에 할당(역할처럼 고르는 게 아님).
- 성향은 긍정/부정/중립 3종(`E_TraitType`, 기존).
- 역할 리액션과 **똑같이** 편집 가능한 부분(관찰 대상·행동 스킬)이 있을 수도, 없을 수도.

---

## 출발점 / 핵심 결정

`Trait` 은 `97.Moon/V1/Trait.cs` 의 평범한 `[Serializable]` 클래스(Type/Name/Description)였고, `BaseCharacterBuilder` 에서 `new Trait(){...}` 로 **코드 생성**됐다. 주석에 `// public Reaction Reaction; // 성향에 고정된 리액션` 이 남아 있어 "성향이 자기 리액션을 가진다"는 의도는 원래 있었음.

리액션을 1:1로 붙이려면 저장/저작 방식을 먼저 정해야 했고, 사용자와 정렬:

- **트레잇을 데이터(SO 카탈로그)로 전환** 채택. "성향을 추가할 때마다 코드를 고치는 건 안 된다"는 이유. (대안: 코드 트레잇 유지+SO 참조 / 트레잇에 인라인 내장 — 모두 기각.)
- 리액션 내용은 **역할과 동일한 `ReactionDefinitionSO`** 재사용(편집 슬롯·깊은복제·완성 판정 그대로).

### 세이브 제약이 데이터 모델을 결정

세이브(`SaveLoadSystem`)는 Newtonsoft JSON + `TypeNameHandling.Auto` 로 `BaseCharacter` 전체를 값 직렬화한다. `Trait[]` 을 `TraitSO[]`(SO 참조)로 바꾸면 SO를 값으로 직렬화하려다 깨짐(에셋 정체성 상실, 역직렬화 실패). → **캐릭터는 트레잇을 Id 문자열로 저장하고 런타임에 카탈로그에서 해석**하는 방식으로 결정.

`.Traits` 접근부가 빌더·Presenter 2곳뿐(배틀은 `TraitReactions` 경유라 무관)이라 변경 blast radius는 좁았음.

## 1단계 — TraitSO 데이터 레이어

- `Assets/02. Scripts/DBSO/TraitSO.cs` (`: BaseSO`): `E_TraitType Type` + `ReactionDefinitionSO ReactionDefinition`(성향↔리액션 1:1). Id/DisplayName/Description 은 BaseSO.
- `Assets/02. Scripts/DBSO/TraitDataBaseSO.cs` (`: DataBaseSO`): "DBSO" 라벨용 빈 서브클래스.

## 2단계 — 트레잇 정체성 마이그레이션

- **`BaseCharacter`**: `public Trait[] Traits` → **`public string[] TraitIds`**(세이브 안전).
  - `GetTrait(int)` — `DBSORegistry.GetSO<TraitSO>(TraitIds[i])` 해석(미설정/미로드면 null).
  - `SyncTraitReactions()` — 보유 성향에 맞춰 `TraitReactions[]` 동기화. 성향↔리액션 1:1. **같은 정의면 기존 런타임 리액션(편집값 포함) 보존**, 달라졌을 때만 새 인스턴스화. Id는 있으나 미해석(미로드)이면 보존(로드 타이밍 유실 방지).
- **`Trait.cs`**: 평범 `Trait` 클래스 제거, `E_TraitType` enum 만 남김.
- **`BaseCharacterBuilder`**: 하드코딩 `List<Trait>` / `SetTraits` 제거. `AddRandomTrait` 이 `DBSORegistry.QuerySO<TraitSO>` 카탈로그에서 **중복 없이** 부여(TraitIds에 Id 기록). `Build()` 끝에 `SyncTraitReactions()`. (StageCoach 가 호출.)
- **`CharacterDetailPresenter.RefreshTraits`**: `TraitIds` 순회 → `GetTrait(i)` 로 TraitSO 해석해 표시(Name→DisplayName).
- **`TownScope`**: 가드된 프리로드 키에 `"TraitDataBase"` 추가.

## 3단계 — 트레잇 리액션 슬롯 UI (슬롯 2-5)

역할 슬롯(0-1)과 **동형 구조**(`slot-main` = 리액션 헤드라인 + 편집행)로 통일. 단:

- **카탈로그 picker 없음** — 리액션은 트레잇이 고정. reaction-part 클릭 핸들러 미등록, 편집 빈칸(관찰대상/행동)만 클릭.
- 편집은 **기존 slotIndex 범용 컨트롤러 메서드 그대로 재사용**(`GetSlotDefinition`/`HasObserveEditable`/`HasSkillEditable`/`GetObserveTargetCandidates`/`SetObserveTarget`/`GetActionSkillCandidates`/`SetActionSkill`/`IsSlotComplete` — 전부 2-5 동작).
- 트레잇 리액션 없으면 "—" + locked, 있으면 정의 DisplayName + 편집슬롯 유무에 따라 빈칸, echo=`IsSlotComplete`.
- 편집 가능 조건을 역할·트레잇 공통(`ApplyViewMode` `editable = fullEdit`) — FullEdit(마을/캠프)에서만.
- `CharacterDetailController.SetCharacter` 가 `SyncTraitReactions()` 호출(표시 전 동기화). `Presenter.RefreshAll` 도 트레잇 렌더 직전 동기화.
- 죽은 구 코드 제거: 컨트롤러 `SetReactionSkill`/`GetAvailableSkills`, Presenter `BuildSkillChips`/`SetTraitPartsVisible`/`GetTriggerDisplayText`/`GetSkillDisplayText`/`_reactionTriggerParts`·`_reactionTriggerLabels`.

## 짚어둘 점

### PartyController → ExpeditionPayload (사용자 리팩토링 반영)
작업 중 사용자가 PartyController를 `ExpeditionPayload` 로 옮기고 RootScope 주입으로 바꿨음(TownScope의 `builder.Register<PartyController>` 주석 처리). `ExpeditionPayload : IPartyController` 이고 `RootScope` 가 `AsImplementedInterfaces()` 로 등록하므로, 자식 TownScope에서 `IPartyController` 가 그대로 해소됨 → CharacterDetailController 의 IPartyController 주입 무영향. 관찰대상 후보 = 실제 탐사 파티(`ExpeditionPayload.GetCurrentParty()`).

### Show() 경로 변경
`CharacterDetailPresenter.Show()` 가 `SetCharacterInternal` 직접 호출 → `_controller.SetCharacter()` 경유로 변경. 컨트롤러 `_character` 가 동기화돼야 리액션 카탈로그/편집이 올바른 캐릭터로 동작하므로(selector 안 거치고 직접 여는 경로 대비) 타당. SyncTraitReactions 도 이 경로로 보장.

## Unity 에디터 수동 작업 (런타임 동작에 필수)

1. `Create → Bond/DBSO/TraitDataBase` 에셋 → **Addressable 주소 정확히 `TraitDataBase`** + 라벨 `DBSO`.
2. `Create → Bond/DBSO/Trait` 에셋 저작: 식별자/표시이름/설명/Type + **ReactionDefinition 연결** → TraitDataBase 의 So List 에 추가.
3. 트레잇이 참조하는 ReactionDefinition 은 `ReactionDefinitionDataBase` 에도 포함돼야 편집 UI(`GetSlotDefinition`)가 해석 가능.
4. 파티(ExpeditionPayload)에 멤버가 있어야 관찰대상 후보 표시.

에셋 없으면 카탈로그/트레잇이 비어도 Town 정상 동작(graceful).
