# 리액션 에디터 — 정의 SO 카탈로그 + 역할 슬롯 UI 재작업

날짜: 2026-05-29
브랜치: Feature/Reaction
선행: `[일일 작업 일지]2026-05-28 리액션 시스템 재구조화.md` 의 "후속 작업" 1·2번

---

## 목표

캐릭터 상세(CharacterDetail)에 들어갈 "리액션 에디터" 구현.

- 역할을 부여하면 역할 리액션 슬롯에서 **역할에 맞는 리액션을 택1**.
- 같은 리액션을 여러 슬롯에서 중복 선택 불가.
- "[지정 아군]이 공격받을 때 [지정 방어 스킬] 사용" 처럼 리액션의 **편집 가능한 부분(관찰 대상·행동 스킬)** 만 플레이어가 채움. 반드시 채워야 작동.
- 마을/탐사/전투 상황에 따른 편집 가능/불가.
- 개발 단계에서 디자이너가 리액션을 만들고, 편집 가능/고정 부분과 스킬 할당 조건을 지정할 수 있어야 함.

05-28 작업에서 런타임 발화(Phase/ObserveFilter/Trigger/Effect, Intercept 등)는 끝나 있었음. 이번엔 그 위에 **"선택 가능한 리액션 카탈로그(데이터·저작)" + "플레이어 편집 UI"** 를 올리는 작업.

---

## 진행 방식 / 결정

사용자와 정렬 후 진행:

1. **데이터·저작 레이어 먼저, UI는 후속** — 카탈로그가 선행돼야 UI 설계가 명확. (그 뒤 "마저 진행" 지시로 UI까지 완료)
2. **신규 DBSO** 로 카탈로그 구성 — `ReactionDefinitionSO : BaseSO` + `ReactionDefinitionDataBaseSO`. `DBSORegistry.QuerySO` 로 역할별 조회 (§6 컨벤션 정렬).
3. **인스펙터 기반 저작** — SO 에셋 + 기존 `[SerializeReference, SubclassSelector]` 인스펙터. 전용 EditorWindow는 만들지 않음.

기존 `ReactionPreset.cs` 는 정의만 있고 어디서도 참조되지 않는 stale 스텁이라 마이그레이션 리스크 없음 → 신규 타입으로 진행, ReactionPreset 은 일단 그대로 둠.

## 1단계 — 데이터·저작 레이어

### 깊은 복제(Clone) 인프라

정의(템플릿)로부터 캐릭터별 런타임 `Reaction` 을 만들 때 Trigger/Effect/조건이 캐릭터 간 공유 참조가 되면 안 됨 → 깊은 복제 도입.

- `Reaction.Clone()` — 값 필드 복사 + `Effect?.Clone()` + `Trigger`(Trigger 타입이면) `.Clone()`. `ITrigger` 인터페이스에는 Clone을 넣지 않고(불필요한 breaking-change 회피) Reaction 안에서 캐스팅으로 처리. 현재 ITrigger 구현은 `Trigger` 뿐.
- `ReactionEffect.Clone()` 추상 + `SkillCastReactionEffect`/`InterceptReactionEffect` 구현.
- `Trigger.Clone()` — Essential/Additional 조건 리스트를 각 `ICondition.Copy()` 로 독립 복사. (GetHitTrigger 등 서브클래스를 Clone하면 동일 조건을 가진 base Trigger가 나옴 — 런타임은 Conditions만 읽으므로 동작 동일, 구체 타입은 비보존. 허용.)
- 조건 `Copy()` 정확성 보강(기존 메서드는 어디서도 호출되지 않아 안전): `SkillTypeCondition.Copy()` 리스트 참조 공유 → 복사, `DamageCondition.Copy()` 가 `Threshold` 를 누락하던 것 → `ThresholdValue.Copy()`(추상+3구현) 추가해 깊은 복사.

### 정의 SO

- **`Assets/02. Scripts/DBSO/ReactionDefinitionSO.cs`** (`: BaseSO`, namespace Reactions)
  - `RoleType Role` — 이 리액션을 선택할 수 있는 역할.
  - `Reaction Template` — Phase/ObserveFilter/Trigger/Effect 가 저작된 고정 골격. 편집 슬롯이 채울 값(SubjectCharacterId, SkillIndex)은 비워 둠.
  - `List<ReactionEditableSlot> EditableSlots` — 플레이어가 채우는 빈칸.
  - `CreateRuntimeReaction()` — Template 깊은 복제 → `DefinitionId = Id` stamp → 각 편집슬롯 `Clear()`.
  - `AllEditablesFilled(reaction)` — 모든 편집 빈칸이 채워졌는지(작동 가능 여부).
  - `OnValidate`(에디터 전용) — 저작 실수 경고: Phase=None / 관찰대상 슬롯인데 ObserveFilter≠Specific / 행동스킬 슬롯인데 Effect≠SkillCastReactionEffect.
- **`Assets/02. Scripts/DBSO/ReactionDefinitionDataBaseSO.cs`** (`: DataBaseSO`) — `[CreateAssetMenu]` 빈 서브클래스. "DBSO" 라벨 부착 대상.

### 편집 슬롯

- **`Assets/02. Scripts/ReactionSystem/ReactionEditableSlot.cs`** (namespace Reactions)
  - 추상 `ReactionEditableSlot` — `Label`, `IsFilled(reaction)`, `Clear(reaction)`, `Description`.
  - `ObserveTargetEditableSlot` — `ExcludeSelf`. `Apply(reaction, allyId)` → `SubjectCharacterId` 채움. (후보=파티 아군 열거는 로스터 필요 → UI 레이어 책임.)
  - `ActionSkillEditableSlot` — `AllowedTypes`(기본 `[DEFENSIVE]`). `ResolveCandidates(reactor)` = 장착 스킬 중 타입 제약 통과분 `(index, skill)`. `Apply(reaction, skillIndex)` → `SkillCastReactionEffect.SkillIndex` 채움.

### 런타임 데이터

- `Reaction.DefinitionId`(string) 추가 — 유래한 정의 Id. 슬롯 간 중복 방지/표시/재조회용.

## 2단계 — 플레이어 UI (역할 슬롯)

역할 슬롯(0-1)만 "리액션 택1 → 빈칸 채우기" 신모델로. **성향(트레잇) 슬롯(2-5)은 요청 범위 밖이라 구조·동작 그대로 보존** → 결과적으로 Presenter에 역할(신)/트레잇(구) 두 경로 공존.

### 컨트롤러 `CharacterDetailController`

- `IPartyController` 주입 → 관찰 후보 = `GetCurrentParty()`.
- 신규: `GetRoleReactionCatalog(slot)`(역할 매칭 + 다른 역할 슬롯에서 쓰인 `DefinitionId` 제외 = **중복 차단**), `SelectRoleReaction`/`ClearRoleReaction`, `GetSlotDefinition`(DefinitionId 역조회), `GetObserveTargetCandidates`/`SetObserveTarget`, `GetActionSkillCandidates`/`SetActionSkill`, `IsSlotComplete`, `HasObserveEditable`/`HasSkillEditable`.
- `SetRole` 의 슬롯 정리를 정의-역할 기반으로 교체. 단 **미로드 시 데이터 유실 방지**: DefinitionId가 비었거나 정의를 못 찾으면 보존하고, 정의를 확실히 찾았는데 역할이 다를 때만 비움.
- 구모델 전용 메서드 4개 제거: `SetReactionTarget`, `SetReactionTrigger`, `GetRoleTriggers`, `GetObserveCandidates()`. (트레잇이 쓰는 `SetReactionSkill`/`GetAvailableSkills` 는 유지.)

### 카탈로그 로드 `TownScope`

- `skillDb` 직접 로드 패턴 옆에 가드된 `DBSORegistry.PreloadSync("ReactionDefinitionDataBase")` 추가. 에셋/Addressable이 아직 없으면 try/catch로 조용히 건너뜀(카탈로그 비활성, Town 정상). 카탈로그 조회는 `DBSORegistry.QuerySO/GetSO` 경유라 미로드 시 빈 결과로 graceful.

### UXML/USS/Presenter

- `CharacterDetail.uxml` — 역할 슬롯 0·1 을 `slot-main`(리액션 헤드라인 행 + 편집행[관찰 대상/행동]) + echo 2행 구조로 교체. 트레잇 슬롯 2-5 및 inline-pool 은 무변경.
- `CharacterDetail.uss` — 규칙 3개만 추가(`slot-main`, `slot-editables`, `slot-part--reaction`). 전부 기존 변수, 1px, 하드코딩/신규 변수 없음 → 스타일 가이드 준수.
- `CharacterDetailPresenter` — 역할 슬롯: 미선택 시 "리액션 선택" + 편집행 숨김, 선택 시 `def.DisplayName` + 편집슬롯 유무에 따라 관찰대상/행동 빈칸 표시. 인라인 풀 분기: `reaction`(카탈로그, "해제" 칩 포함) / `observe`(파티 아군) / `skill`(장착+타입제약). echo = `IsSlotComplete`(◈/○). 편집은 FullEdit 에서만. 트레잇 슬롯 갱신/스킬 풀은 기존 로직 유지.
  - 편집 빈칸이 없는 완전 고정 리액션이면 편집행 자체를 숨김(빈 행 border만 남는 결함 수정).

## Unity 에디터 수동 작업 (런타임 동작에 필요)

코드만으로는 카탈로그가 비어 있음. 다음을 해야 실제로 동작:

1. `Create → Bond/DBSO/ReactionDefinitionDataBase` 에셋 생성.
   - **Addressable 주소를 정확히 `ReactionDefinitionDataBase` 로 설정** (TownScope가 이 키로 PreloadSync). + 라벨 `DBSO`.
2. `Create → Bond/Reactions/Reaction Definition` 에셋 저작:
   - 식별자(Id, 고유) / 표시이름 / Role
   - Template: Phase(PreApply/PostApply), ObserveFilter, Trigger 조건, Effect
   - EditableSlots: 필요 시 관찰대상/행동스킬 슬롯
   - DB 에셋의 **So List** 에 추가
3. 파티에 멤버가 있어야 관찰대상 후보 표시.

### 예시 저작
- *체력 20% 이하 자신 제외 아군 대신맞기* (완전 고정): Role=Tanker, Phase=PreApply, ObserveFilter=OtherAlly, Trigger=[SubjectIs(Target), SkillTypeIs(OFFENSIVE/SPELL), HitCondition, HpBelow(0.2)], Effect=Intercept, EditableSlots=비움.
- *[지정 아군] 피격 시 [지정 방어스킬]* (편집형): Role=Tanker, Phase=PreApply, ObserveFilter=Specific, Trigger=[SubjectIs(Target), SkillTypeIs(OFFENSIVE/SPELL), HitCondition], Effect=SkillCast(SkillTarget=Observed, SkillIndex 비움), EditableSlots=[ObserveTarget, ActionSkill(DEFENSIVE)].
