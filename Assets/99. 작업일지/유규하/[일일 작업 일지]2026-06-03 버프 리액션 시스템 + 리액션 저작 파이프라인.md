# 2026-06-03 — 버프 리액션 시스템 + 리액션 저작 파이프라인

한 줄 요약: 전투 버프/디버프 시스템을 새로 구현하고(서포터 공격력/방어력 버프 리액션 가능해짐), 역할·성향 리액션을 코드로 찍어내는 저작 파이프라인(플루언트 빌더 + 배관 + 메뉴 생성기)을 만들었다. 파이프라인 스텝 재배치와 HpRatio 버그도 같이 처리.

---

## 1. 출발점 조사 결론

- **기존에 버프/디버프/상태이상은 구현돼 있지 않았다.**
  - `BuffEffect`(스킬 효과), `StatusEffectEventEffectHandler`(맵 이벤트) 둘 다 빈 stub(TODO 로그만).
  - 유일하게 동작하던 `StatModifier` / `StatController`는 **장비(악세서리) 스탯 보정 + 직업/몬스터 기본 스탯 계산 전용**. duration 개념 없음 → 턴제 버프로는 못 씀.
- 리액션 시스템으로 표의 6개 역할 리액션 판정:
  - 딜러 회피반격 / 딜러 치명타추격 / 탱커 엄호 / 탱커 위기엄호 → **기존 타입 조합으로 표현 가능**.
  - 서포터 공격력버프 / 방어력버프 → 트리거는 표현 가능하나 **버프 "행동"이 부재** → 이번에 구현.
- 핵심 구조 파악:
  - **Stat은 baked.** `Profession.CalculateStat`이 base+장비+`StatController`로 `atk/def/Sp_Atk` 등을 다시 구워 `Stat`에 씀. `current_Hp`는 안 건드림 → 전투 중 재계산 안전.
  - 악세서리가 `StatController.AddModifiers` → `CalcStat()` 패턴으로 동작 → **버프 = "아이템 없는 악세서리 + 턴 타이머"**.
  - 파이프라인 스텝 순서(asset): `Entry → [PreApply] → Evasion → Critical → Defense → Apply → [PostApply]`.

## 3. 설계 결정 (왜 이렇게 했나)

- **공격력↑ = `DamageMultiplier`, 방어력↑ = `DamageReduction`** 사용.
  - 이 두 StatType은 원래 아무 코드도 안 읽던 inert 상태 → 기존 동작 안 깨고 파이프라인에 새로 꽂음.
  - `atk/def`(시트 스탯)보다 "데미지 증감 비율"이 버프 의미에 더 맞음.
- **파이프라인이 라이브로 읽음** → 버프 적용/만료가 `Add/RemoveModifiers`만으로 끝나고 `CalcStat()` 재굽기 불필요.
- **base 1 트릭**: `ApplyModifiers(type, 1) - 1`로 읽음.
  - base 0으로 읽으면 Percent 모디파이어가 0에 곱해져 사라지는 함정이 있었음(=Flat 강제).
  - base 1로 읽고 1 빼면 `(1+ΣFlat)(1+ΣPercent)-1` → **Flat/Percent 둘 다 추가비율로 합산**. mode 무관, 음수면 취약(디버프)으로 동작.
- **지속 = 버프 받은 캐릭터의 자기 턴 수** (라운드 아님). tick은 `TurnManager` 유닛 루프 1곳.
- **버프 값은 인라인**(BuffReactionEffect.Modifiers). 추후 `StatModifierDataSO` 카탈로그로 마이그레이션 가능.

## 4. 파이프라인 스텝 재배치 (PreApply를 Entry 앞으로)

- 변경: `Assets/03. PipeLine/BattlePipeLineSO.asset`의 `steps`에서 ReactionCall[PreApply]를 EntryStep **앞**으로 이동.
  - 새 순서: `[PreApply] → Entry → Evasion → Critical → Defense → Apply → [PostApply]`.
- 이유: 서포터 공격력 버프가 **트리거된 그 공격부터** 실리게 하려면, 버프가 EntryStep(=value 산출) 전에 caster에 걸려야 함.
- 안전성 근거:
  - PreApply에서 `context.value`를 읽는 조건은 `DamageCondition` 하나뿐인데, **이걸 쓰는 리액션 에셋이 0개**(전수 확인).
  - `EntryStep`은 caster.Stat+스킬만 읽고 **target은 안 읽음** → Intercept(target 교체)와 무관, value 동일.
  - Evasion/Critical/Defense/Apply는 전부 Entry 뒤 그대로 → 영향 없음.
- 남는 제약 1개: **`DamageCondition`은 이제 PostApply 전용**으로 간주(PreApply에선 value=0). 의미상으로도 맞음.
- `InterceptReactionEffect` 주석을 "Evasion/Critical/Defense 이전" 기준으로 정정.

## 6. 리액션 저작 파이프라인 (코드로 리액션 찍어내기)

리액션은 고정 어휘(조건 8 / 행동 3 / 슬롯 2)의 **조합**이라, 어휘를 아는 빌더면 어떤 역할·성향 리액션이든 데이터로 생성 가능. (새 "부품 타입"이 필요할 때만 코드 확장.)

- `Assets/02. Scripts/ReactionSystem/Trigger.cs`
  - `WithConditions(params ICondition[])` 추가 — 조건을 인스펙터-가시(Additional) 목록에 채움.
- `Assets/Editor/Reactions/ReactionDefBuilder.cs` (Editor)
  - 플루언트 빌더. `Def().Name().Phase().Observe().When(...).Do(...).Editable(...).Build(folder)`.
  - 단축 팩토리: `SubjectIs/SkillTypeIs/HpBelow/Crit/Evaded/Hit/Killed/DamageAtLeast*` (조건), `Buff/Intercept/CastSkill` (행동), `ObserveTarget/ActionSkill` (슬롯).
  - `Validate()`가 ReactionDefinitionSO.OnValidate 규칙(Phase≠None, ObserveTarget슬롯↔Specific, ActionSkill슬롯↔CastSkill)을 빌드 시 선검사.
- `Assets/Editor/Reactions/ReactionAuthoringIO.cs` (Editor)
  - 공유 배관: 폴더 보장 / BaseSO `_id` SerializedObject 주입 / 에셋 생성·교체(있으면 `CopySerialized`로 제자리 덮어써 **GUID 보존** → 재실행 안전) / DB `_soList` 등록(Id 기준 중복 교체 + null 정리).
- `Assets/Editor/Reactions/ReactionCatalogGenerator.cs` (Editor)
  - 메뉴 `Bond/Reactions/역할 리액션 카탈로그 생성` → 6종 저작 + DB 생성/등록.
  - 생성물: `Assets/Data/Reactions/RACT_*.asset` + `ReactionDefinitionDataBase.asset`.

### 생성되는 6종
1. `RACT_DEAL_COUNTER_EVADE` 회피 반격 — Dealer/PostApply/Self, When(SubjectIs(Target),Evaded) → CastSkill(Caster), 편집:ActionSkill(OFFENSIVE/SPELL)
2. `RACT_DEAL_COUNTER_CRIT` 치명타 추격 — Dealer/PostApply/Self, When(SubjectIs(Caster),Crit) → CastSkill(Target), 편집:ActionSkill(OFFENSIVE/SPELL)
3. `RACT_TANK_COVER` 엄호 — Tanker/PreApply/Specific, When(SubjectIs(Target),SkillType(OFF,SPELL)) → Intercept, 편집:ObserveTarget
4. `RACT_TANK_COVER_LOWHP` 위기 엄호 — Tanker/PreApply/OtherAlly, When(SubjectIs(Target),SkillType(OFF,SPELL),HpBelow(0.2)) → Intercept
5. `RACT_SUP_ATKUP` 지원 공격 강화 — Supporter/PreApply/Specific, When(SubjectIs(Caster),SkillType(OFF)) → Buff(DamageMultiplier,0.3,2턴), 편집:ObserveTarget
6. `RACT_SUP_DEFUP` 지원 방어 강화 — Supporter/PreApply/Specific, When(SubjectIs(Target),SkillType(OFF,SPELL)) → Buff(DamageReduction,0.2,2턴), 편집:ObserveTarget

### 어휘 확장 방법 (예: "아군 사망 시" 새 조건)
1. `Condition.cs`에 `AllyDeadCondition : ReactionTriggerCondition` 추가
2. `ReactionDefBuilder`에 `public static ICondition AllyDead() => new AllyDeadCondition();` 한 줄
→ 이후 모든 리액션이 `AllyDead()`를 데이터로 사용.
