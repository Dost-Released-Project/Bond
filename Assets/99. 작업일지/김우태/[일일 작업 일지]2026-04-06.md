## 1. 작업 개요

> VContainer 스코프 이슈 해결 및 다키스트 던전 스타일의 진영(Formation) 시스템 고도화

- **주요 작업 영역 :** BattleSystem, VContainer Configuration, Formation Logic
- **작업 목표 :** 
    - VContainer의 Scope 간 인스턴스 불일치 문제 해결
    - `BattleSlot` 도입을 통한 진영 관리 로직의 객체지향적 리팩토링

- **작업 결과 요약 :** 
    - `BattleFlowManager`와 `TurnUI`가 서로 다른 `TurnManager` 인스턴스를 참조하던 문제를 `Lifetime.Singleton` 설정으로 해결.
    - 기존 `BaseCharacter[]` 기반 관리 방식을 `BattleSlot` 객체 관리 방식으로 전환하여 위치 기반 효과 확장이 용이해짐.
    - 불필요한 `Key` 데이터 클래스를 제거하고 `BattleSlot`을 직접 반환하도록 인터페이스(`IFormationManager`) 개선.

## 2. 문제 사항 / 변경 사항 / 주요 성과

### 📌 이슈 #1 : VContainer Scope 간 인스턴스 공유 오류

#### 문제 사항
- **문제 설명 :** `TurnLifeTimeScope`와 그 하위 스코프에서 `TurnManager`를 `Lifetime.Scoped`로 등록했을 때, 각 스코프마다 새로운 인스턴스가 생성되어 `TurnUI`와 `BattleFlowManager`가 서로 다른 데이터를 참조하는 현상 발생.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | `Lifetime.Scoped`로 등록되어 부모/자식 스코프 간 인스턴스가 분리됨 |
| **After** | `Lifetime.Singleton`으로 변경하고 인터페이스 주입(`AsImplementedInterfaces`)을 통해 단일 인스턴스 공유 보장 |

```csharp
// Before
builder.Register<TurnManager>(Lifetime.Scoped).AsImplementedInterfaces();

// After
builder.Register<TurnManager>(Lifetime.Singleton).AsImplementedInterfaces().AsSelf();
```

#### 주요 성과 및 해결 이유 (Why)
- **선택 이유 :** 전투 전반에서 상태를 공유해야 하는 매니저 객체는 해당 생명주기 내에서 단일 인스턴스임이 보장되어야 함.
- **최종 선택 근거 :** VContainer에서 상위 스코프의 객체를 하위 스코프와 공유하기 위해서는 `Singleton` 레벨의 등록이 가장 안정적이며, 명시적인 인터페이스 주입을 통해 의존성 결합도를 낮춤.

### 📌 이슈 #2 : 진영 관리 시스템(Formation)의 구조적 한계

#### 문제 사항
- **문제 설명 :** 기존 `BaseCharacter[]` 배열 방식은 빈 자리 처리나 위치 자체의 속성(함정, 장판 등)을 관리하기 어렵고, `Key` 객체를 통한 잦은 데이터 변환으로 인해 코드가 복잡함.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | `BaseCharacter[4]` 배열을 직접 스왑하며, 위치 정보를 `Key` 클래스로 변환하여 전달 |
| **After** | `BattleSlot` 클래스를 도입하여 위치 정보를 객체화하고, 캐릭터는 슬롯의 `Occupant`로 관리 |

```csharp
// Before: 데이터 변환 필요
public Key GetFormationKey(BaseCharacter character) {
    // 인덱스 찾아서 new Key 생성 후 반환
}

// After: 슬롯 직접 참조
public BattleSlot GetSlot(BaseCharacter character) {
    return _Slots.Values.SelectMany(s => s).FirstOrDefault(s => s.Occupant == character);
}
```

#### 주요 성과 및 해결 이유 (Why)
- **선택 이유 :** 다키스트 던전 스타일의 '열(Rank)' 시스템은 위치 자체가 데이터(상태)를 가져야 하므로 객체지향적인 `Slot` 구조가 필수적임.
- **최종 선택 근거 :** `BattleSlot` 도입으로 인해 `SwapFormation` 시 데이터 손실 위험이 줄어들고, 향후 특정 위치에 버프/디버프 장판 효과를 추가하기 위한 기반을 마련함.

## 3. 상세 시스템 사용 가이드

---

### 🛠️ Formation & Slot 시스템 사용 가이드

#### 개요
- **대상 :** 전투 시스템 개발자 및 스킬 기획자
- **목적 :** 캐릭터의 위치 정보를 조회하고, 스킬 사용/타겟팅 가능 여부를 비트마스크로 판정

#### 사용 방법

**Step 1. 캐릭터 슬롯 조회**  
```csharp
// IFormationManager를 주입받아 사용
var slot = formationManager.GetSlot(myCharacter);
Debug.Log($"현재 위치: {slot.Side} 진영의 {slot.Rank}열");
```

**Step 2. 스킬 사용 가능 여부 체크**  
```csharp
// 스킬 데이터에 정의된 FormationMask(예: Form3 | Form4)와 비교
if (formationManager.Calc_SkillUsable(myCharacter, skillData.UsableMask)) {
    // 스킬 활성화
}
```

**Step 3. 캐릭터 위치 교체 (Swap)**  
```csharp
// 두 캐릭터의 위치를 안전하게 교체
formationManager.SwapFormation(characterA, characterB);
```

#### 주의 사항
- ⚠️ `BattleSlot`의 `Side`와 `Rank`는 생성 시 결정되는 고정 값입니다. 캐릭터의 이동은 `Occupant` 속성 변경을 통해 이루어집니다.
- ⚠️ `GetSlot` 호출 시 캐릭터가 전장에 존재하지 않으면 `null`을 반환하므로 Null 체크가 필요합니다. (`slot?.Rank ?? FormationMask.None` 활용 권장)
