### 📌 이슈 #1 : 파이프라인 중단 점(Breakpoint) 및 에디터 구성 기능 추가

#### 문제 사항
- **문제 설명 :** 기존 전투 로직이 하나의 메서드에 몰려 있어, 회피 발생 시 이후 데미지 계산을 스킵하는 등의 분기 처리가 복잡해지고 가독성이 떨어짐. 또한 새로운 전투 효과 추가 시 코드를 매번 수정해야 함.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | `BattleManager` 내에서 모든 전투 로직이 순차적으로 실행되며 조건문으로 분기 처리 |
| **After** | `PipeLineSo` 시스템을 도입하여 `Step` 단위로 로직 분리 및 `ShouldBreak`를 통한 중단 제어 |

```csharp
// After: PipeLineSo.cs 의 핵심 실행 루프
foreach (var step in steps)
{
    context = step.Execute(context);
    if (ShouldBreak(context)) // 중단 조건(예: 회피 성공) 체크
    {
        Debug.Log($"Pipeline broken at step: {step.GetType().Name}");
        break;
    }
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** **유연한 확장성 확보**. 전투 로직을 `ScriptableObject` 기반의 파이프라인으로 분리함으로써, 기획자가 코딩 없이 에디터에서 순서를 바꾸거나 새로운 단계를 추가할 수 있게 함.
- **고려했던 다른 방법 :** 상태 패턴(State Pattern)이나 단순 메서드 체이닝.
- **최종 선택 근거 :** 유니티 인스펙터와의 궁합이 가장 좋고, `SerializeReference`를 활용해 데이터와 로직을 한 곳에서 관리할 수 있는 파이프라인 방식이 프로젝트 스택에 가장 적합하다고 판단함.

### 📌 이슈 #2 : 공용 파이프라인 에디터(`GenericPipeLineEditor`) 구축

#### 문제 사항
- **문제 설명 :** 새로운 파이프라인 타입(예: 인벤토리용, 퀘스트용)을 만들 때마다 전용 커스텀 에디터를 작성해야 하는 번거로움이 발생함.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | `BattlePipeLineSoEditor` 등 특정 클래스에 종속된 에디터 사용 |
| **After** | `PipeLineSoBase`를 타겟으로 하는 제네릭 에디터 구현으로 모든 파이프라인 자동 지원 |

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** **중복 코드 제거 및 유지보수 효율화**. 리플렉션을 통해 파이프라인의 `Context` 타입을 동적으로 찾아내고, 해당 컨텍스트에 맞는 `Step` 목록만 필터링하여 드롭다운으로 보여줌.
- **최종 선택 근거 :** 생산성 향상. 앞으로 어떤 도메인의 파이프라인을 추가하더라도 에디터 작업 없이 즉시 사용 가능함.

### 📌 이슈 #3 : 팀원(이주노) 스크립트(`TurnLifetimeScope`)와의 의존성 주입 연결

#### 문제 사항
- **문제 설명 :** 새로 구축한 파이프라인 시스템이 팀원 이주노가 작성한 `TurnLifetimeScope` 기반의 턴 시스템과 물리적으로 분리되어 있어, 실제 전투 런타임에서 `BattleManager`에 파이프라인이 주입되지 않는 문제 발생.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | `BattleManager`가 파이프라인 없이 비어있는 상태 |
| **After** | `TurnLifetimeScope`에서 `IBattlePipeLine`을 등록하여 `BattleManager`에 자동 주입 |

```csharp
// After: TurnLifetimeScope.cs 연동 코드
[SerializeField] private BattlePipeLineSo skillApplyPipeline;

protected override void Configure(IContainerBuilder builder)
{
    builder.Register<BattleManager>(Lifetime.Scoped);
    if (skillApplyPipeline != null)
    {
        builder.RegisterInstance<IBattlePipeLine>(skillApplyPipeline);
    }
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** **인터페이스 기반 추상화**. `BattleManager`가 구체 클래스가 아닌 `IBattlePipeLine` 인터페이스에 의존하게 하여, 이주노 팀원의 스크립트 수정을 최소화하면서 시스템을 통합함.

## 3. 상세 시스템 사용 가이드

---

### 🛠️ 전투 파이프라인(BattlePipeLine) 사용 가이드

#### 개요
- **대상 :** 전투 로직 설계자 및 개발자
- **목적 :** 전투 계산 및 연출 단계를 에디터에서 구성

#### 사용 방법

**Step 1. 파이프라인 에셋 생성**  
Project 창에서 우클릭 후 `Create > PipeLine > BattlePipeLineSO`를 선택하여 에셋을 생성합니다.

**Step 2. 단계(Step) 구성**  
생성된 에셋의 인스펙터에서 `+` 버튼을 눌러 `EntryStep`, `EvasionStep` 등을 순서대로 추가합니다.

**Step 3. 턴 시스템에 할당**  
씬에 있는 `TurnLifetimeScope` 오브젝트를 선택하고, `Skill Apply Pipeline` 슬롯에 생성한 에셋을 드래그하여 할당합니다.

#### 주의 사항

- ⚠️ **에셋 미할당 주의**: `TurnLifetimeScope`에 에셋을 할당하지 않으면 `BattleManager`에서 파이프라인을 실행할 수 없습니다.
- ⚠️ **네임스페이스**: 현재 모든 파이프라인 관련 클래스는 `_03._PipeLine` 네임스페이스를 사용합니다.
