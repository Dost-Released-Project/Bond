# [일일] 2026-04-08

## **1. 요약**
전투 진영 관리 시스템(`FormationManager`)을 **DLV 아키텍처(Data-Logic-Visual)**로 리팩토링하여 데이터 보존, 판단 로직, 시각 연출의 역할을 명확히 분리함.

---

## **2. 문제 및 변경 사항**

### **[Before]**
*   `FormationManager`가 진영 상태(`BattleSlot[]`)를 직접 관리하고 수정함. (로직-데이터 혼재)
*   비주얼 연출을 위한 연결 고리가 부재(`TODO` 주석 상태).
*   판단 로직이 데이터 직접 수정에 의존하여 유연성이 낮음.

### **[After]**
*   **[D] Runtime Data 분리:** `RuntimeDataFormation.cs` 신규 생성. 슬롯 상태 관리 및 이벤트(`Observable`) 발송 담당.
*   **[L] Logic System 개편:** `FormationManager.cs`가 데이터 직접 수정 대신 판단 후 `RuntimeData` 및 `Visualizer` 호출.
*   **[V] Visual Interface 정의:** `IFormationVisualizer.cs` 신규 생성. 로직이 구체적인 연출 방식(2D/3D 등)을 몰라도 명령을 내릴 수 있도록 DIP 적용.

---

## **3. 주 의사결정 근거 (Why)**
*   **데이터 오염 방지:** 로직이 데이터를 직접 수정하면 추적이 어렵고 오류가 발생하기 쉬움. `RuntimeData`를 통해 모든 변경을 캡슐화하고 이벤트를 통해 상태 변화를 통지하도록 함.
*   **결합도 해제:** 연출 방식(Visual)이 바뀌더라도 로직(Logic)을 수정할 필요가 없도록 인터페이스로 추상화함.
*   **가이드라인 준수:** 프로젝트의 핵심 철학인 "입력은 흐르고, 데이터는 보존되며, 로직은 판단하고, 비주얼은 표현한다"를 충실히 따름.

---

## **4. 테스트 가이드 및 연출 가이드**
*   **테스트:** `FormationManager` 생성 시 `RuntimeDataFormation` 인스턴스가 정상 주입되었는지 확인.
*   **연출 연결:** 진영 연출용 스크립트에서 `IFormationVisualizer`를 상속받아 `MoveCharacter`, `SwapFormation` 호출 시 연출이 정상적으로 호출되는지 로그 또는 디버그 체크.
