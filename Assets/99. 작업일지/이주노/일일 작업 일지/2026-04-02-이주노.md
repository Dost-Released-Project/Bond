## 1. 작업 개요

> VContainer와 UniTask를 활용하여 턴제 전투 시스템의 코어 로직과 플레이어 유닛의 턴 대기 처리의 기반을 구현했습니다.

- **주요 작업 영역 :** 턴제 전투 시스템 코어 아키텍처 설계 및 구현
- **작업 목표 :** 의존성 주입(DI)을 통한 전투 진입점 설정 및 속도(Speed) 기반의 비동기 턴 흐름 제어 시스템 구축
- **작업 결과 요약 :** - `TurnLifetimeScope` 및 `BattleEntryPoint`를 통한 전투 초기화 로직 구현
    - `TurnManager`를 이용한 라운드 진행 및 생존 유닛의 턴 큐(Turn Queue) 관리 구현
    - `ITurnUseUnit` 인터페이스 정의 및 `TestPlayer`의 스페이스바 입력 기반 턴 종료 로직 추가 완료

## 2. 문제 사항 / 변경 사항 / 주요 성과

### 📌 이슈 #1 : 턴제 시스템 아키텍처 설계 및 DI(의존성 주입) 적용

#### 문제 사항
- **문제 설명 :** 전투 씬 진입 시 전투 매니저와 씬 내 유닛들 간의 참조 관계를 유연하게 구성하고 관리할 진입점이 필요함.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | 전투 매니저 및 유닛 할당 시스템 부재 |
| **After** | VContainer를 활용한 `TurnLifetimeScope` 및 `BattleEntryPoint` 구현으로 시스템 결합도 완화 |

```csharp
// After (핵심 구현 내용)
protected override void Configure(IContainerBuilder builder)
{
    builder.Register<TurnManager>(Lifetime.Scoped);
    builder.RegisterInstance(sceneUnit);
    builder.RegisterEntryPoint<BattleEntryPoint>();
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** VContainer의 `IAsyncStartable`을 활용해 비동기적으로 전투를 시작(`BattleEntryPoint.StartAsync`)함으로써 초기화 시점의 안전성을 확보했습니다.
- **최종 선택 근거 :** 매니저 클래스 간의 강한 결합을 피하고, 추후 단위 테스트 및 모듈 교체가 용이한 구조를 가져가기 위해 VContainer를 채택했습니다.

### 📌 이슈 #2 : 비동기 기반 턴 흐름 및 입력 대기 제어

#### 문제 사항
- **문제 설명 :** 각 유닛의 턴이 순차적으로 진행되어야 하며, 특히 플레이어 턴의 경우 사용자의 입력이 끝날 때까지 턴 진행이 대기되어야 함.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | 턴 진행 및 대기 로직 부재 |
| **After** | UniTask 및 `AutoResetUniTaskCompletionSource`를 활용한 비동기 입력 대기 로직 구현 |

```csharp
// After (TestPlayer.cs 내 비동기 대기 처리)
public async UniTask TakeTurnAsync()
{
    Debug.Log($"<color=green>{unitName} 차례! 플레이어의 명령을 기다립니다...</color>");
    _tcs = AutoResetUniTaskCompletionSource<bool>.Create();
    await _tcs.Task; // 스페이스바 입력 시 _tcs.TrySetResult(true) 호출로 해제
    _tcs = null;
    Debug.Log($"{unitName} 행동 완료!");
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** 유니티 코루틴(Coroutine) 대신 가비지 할당(GC Allocation)이 적고 반환값 처리가 명확한 `UniTask`를 사용했습니다.
- **최종 선택 근거 :** 플레이어의 입력을 이벤트(`InputAction`)로 받고, 이를 `AutoResetUniTaskCompletionSource`와 결합하여 자원 소모 없이 대기 상태를 유지하도록 설계했습니다.
