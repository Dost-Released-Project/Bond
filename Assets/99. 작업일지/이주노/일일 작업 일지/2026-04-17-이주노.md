**MapGenerator 7단계 알고리즘:**

```csharp
public MapData GenerateMap(int seed, int actNumber)
{
    System.Random rng = new System.Random(seed);  // 시드 고정 → 재현성 보장

    // Step 1: 층별 노드 수 결정 (0층·마지막층 = 1개 고정)
    int[] nodeCounts = DetermineNodeCounts(rng, data.TotalLayers);

    // Step 2: 노드 생성 + Fisher-Yates 셔플로 열 배치
    CreateNodes(data, nodeCounts, rng);

    // Step 3: ±1열 범위 내 연결 + 고립 노드 강제 연결
    ConnectNodes(data, rng);

    // Step 4: 교차 엣지 감지 → 스왑으로 제거 (수렴할 때까지 반복)
    RemoveCrossings(data);

    // Step 5: 층별 가중치 테이블로 스테이지 타입 배정
    AssignStageTypes(data, rng);

    // Step 6: 보스·캠핑 강제 배치, Elite 최소 1개, Camping 최소 2개 보장
    ApplyPlacementRules(data, rng);

    // Step 7: 전체 Locked → 0층만 Available
    InitializeNodeStates(data);

    return data;
}
```

**층별 스테이지 타입 가중치 테이블:**

| 층 조건 | Normal | Elite | Event | Camping | Shop |
|---------|--------|-------|-------|---------|------|
| layer == 0 | 100% | - | - | - | - |
| layer < 4 | 70% | - | 30% | - | - |
| layer < EliteMinLayer | 55% | - | 25% | 15% | 5% |
| 일반 중간층 | 45% | 15% | 22% | 12% | 6% |
| preBossLayer | - | - | - | 100% | - |
| lastLayer | - | - | - | - | Boss |

**MapNavigator 상태 전환 규칙:**

```
노드 클릭
  → MoveToNode(nodeId)
     → 이전 CurrentNode → Visited
     → 이전 CurrentNode.NextNodeIds 전체 → Locked (선택 안 한 경로 차단)
     → 선택한 노드 → Current
     → 선택한 노드.NextNodeIds → Available
     → MapRepository.Save()
     → OnNodeEntered 이벤트 발생
```

**MapData 직렬화 구조:**

```csharp
    public int CurrentNodeId;            // JSON 직렬화됨
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 (DAG 구조):** 순환 경로가 없는 방향 그래프로 설계해 플레이어가 항상 위층으로만 이동하도록 강제했다. Slay the Spire와 동일한 구조로 진행 방향이 명확하다.
- **선택 이유 (Seed 결정론):** `System.Random(seed)`을 하나만 생성해 전체 생성 과정에서 공유한다. 같은 시드면 항상 동일한 맵이 재현되어 디버그와 저장/불러오기가 안전하다.
- **선택 이유 (Repository 패턴):** 저장 로직을 `IMapRepository`로 추상화해 현재 JSON 파일 저장 방식을 나중에 클라우드/DB 방식으로 교체해도 나머지 코드를 수정할 필요가 없다.
- **선택 이유 (NonSerialized + BuildLookups):** `JsonUtility`는 Dictionary를 직렬화하지 못한다. List만 직렬화하고 로드 후 `BuildLookups()`로 딕셔너리를 재구성하면 JSON 호환성과 O(1) 조회 성능을 동시에 확보할 수 있다.
- **고려했던 다른 방법 :** Newtonsoft.Json으로 Dictionary 직렬화, 노드 좌표를 월드 좌표로 직접 저장
- **최종 선택 근거 :** `JsonUtility`는 Unity 내장이라 추가 패키지 없이 사용 가능하고, 정규화 좌표(0~1)는 해상도/화면 크기와 무관하게 UI 배치가 가능해 확장성이 높다.

### 📌 이슈 #3 : StageLoader 컴파일 에러 — `SceneAddress` 필드 누락

#### 문제 사항
- **문제 설명 :** `StageLoader.cs`에서 `config.SceneAddress`를 참조하고 있었으나, `StageConfig.cs`에 해당 필드가 없어 `CS1061` 컴파일 에러 발생. Map 시스템 전체가 빌드 불가 상태였다.

```
Assets\98. LeeJuno\Juno_Scripts\Map\Loader\StageLoader.cs(37,20): error CS1061:
'StageConfig' does not contain a definition for 'SceneAddress'
```

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | `StageConfig`에 `IconAddress`만 존재. `StageLoader`에서 `config.SceneAddress` 참조 시 컴파일 에러 |
| **After** | `StageConfig`에 `public string SceneAddress` 필드 추가. 에디터 Inspector에서 씬 Addressable 주소 입력 가능 |

```csharp
// Before

{
    public string IconAddress;
    public Color NodeColor;
    // SceneAddress 없음 → StageLoader에서 참조 시 에러

{
    public string IconAddress;
    public string SceneAddress;   // 추가 — Addressables 씬 주소
```

- **선택 이유 :** 스테이지 타입별로 로드할 씬 주소가 다르므로 `StageConfig` ScriptableObject에 필드로 보유하는 것이 가장 자연스럽다. Inspector에서 타입별로 개별 설정이 가능하다.
- **고려했던 다른 방법 :** `Dictionary<StageType, string>` 하드코딩, 별도 SceneAddressConfig SO
- **최종 선택 근거 :** `StageConfig`가 이미 타입별 시각/설정 데이터를 보유하고 있으므로 씬 주소도 같은 곳에서 관리하는 것이 SSOT(단일 진실 공급원) 원칙에 부합한다.

### 📌 이슈 #4 : Map 폴더 전체 설명 주석 부재

#### 문제 사항
- **문제 설명 :** Map 시스템 21개 스크립트 전체에 설명 주석이 전혀 없었다. 각 클래스의 역할, 메서드의 동작 원리, 주의사항(BuildLookups 호출 타이밍, NonSerialized 이유, Slay the Spire 방식 상태 전환 등)이 코드만으로는 파악하기 어려웠다.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | 21개 파일 모두 주석 0줄. 메서드 이름만으로 동작 유추 필요 |
| **After** | 클래스 요약 주석, 메서드별 동작·파라미터·주의사항 주석 추가. 알고리즘 설명(Fisher-Yates, 교차 제거 조건 등) 포함 |

```csharp
// Before
public bool MoveToNode(int nodeId)
{
    if (_mapData == null) return false;
    MapNode target = _mapData.NodeById[nodeId];
    if (target.State == NodeState.Available == false) return false;
    // ...
}

// After
/// <summary>
/// 지정한 nodeId의 노드로 이동을 시도한다.
/// Available 상태인 노드에만 이동 가능하다.
/// 성공 시 상태를 전환하고, 맵을 저장한 뒤 OnNodeEntered를 발생시킨다.
/// </summary>
public bool MoveToNode(int nodeId)
{
    if (_mapData == null) return false;
    MapNode target = _mapData.NodeById[nodeId];

    // Available 상태가 아니면 이동 불가
    if (target.State == NodeState.Available == false) return false;

    // 이전 노드 정리: Visited 처리 + 선택하지 않은 다음 노드 잠금
    // Slay the Spire 방식: 같은 층의 다른 경로는 영구 잠금
    // ...
}
```

- **선택 이유 :** Map 시스템은 7단계 알고리즘(교차 제거, Fisher-Yates, 가중치 배정 등)과 비직관적인 직렬화 주의사항(`BuildLookups` 호출 타이밍, `[NonSerialized]` 이유)이 존재한다. 팀원이 코드를 읽을 때 이 부분을 주석 없이 파악하는 데 불필요한 시간이 소모된다.
- **고려했던 다른 방법 :** 별도 기술 문서만 작성, README 작성
- **최종 선택 근거 :** 코드와 문서가 분리되면 시간이 지나면서 불일치가 생긴다. 코드 내 주석이 가장 신뢰도가 높고, IDE에서 마우스 오버 시 즉시 확인 가능하다.

### 🛠️ Map 시스템 초기 설정 가이드

#### 개요
- **대상 :** Map 씬을 처음 설정하는 팀원
- **목적 :** MapLifetimeScope Inspector 연결 방법 및 스테이지 씬 Addressable 등록 순서 안내

#### 사전 준비

```
필요 에셋:
- MapGeneratorConfig ScriptableObject (Create → Bond → MapGeneratorConfig)
- StageConfig ScriptableObject × 6 (Normal, Elite, Boss, Camping, Event, Shop 각 1개)
  (Create → Bond → StageConfig)
- 스테이지 씬 파일 (Addressable 등록 완료 상태)
```

#### 사용 방법

**Step 1.** 씬에 빈 GameObject를 만들고 `MapLifetimeScope` 컴포넌트를 추가한다.

**Step 2.** Inspector에서 다음 항목을 연결한다.
```
_generatorConfig  → MapGeneratorConfig ScriptableObject
_stageConfigs     → StageConfig 6개 (타입별 1개씩)
_mapUIController  → 씬에 배치된 MapUIController GameObject
```

**Step 3.** 각 `StageConfig` 에셋을 선택하고 `SceneAddress`에 Addressable 씬 주소를 입력한다.
```
예시)
Normal StageConfig → SceneAddress: "Scenes/NormalStage"
Boss StageConfig   → SceneAddress: "Scenes/BossStage"
```

**Step 4.** `MapUIController`가 있는 GameObject에 `MapView`와 `_mapPanel`(맵 패널 루트 GameObject)을 연결한다.

**Step 5.** `MapView`에 다음 항목을 연결한다.
```
_mapContainer   → 맵 노드·엣지를 배치할 RectTransform
_nodeViewPrefab → MapNodeView 프리팹
_edgeViewPrefab → LineRenderer가 붙은 MapEdgeView 프리팹
_stageConfigs   → StageConfig 6개 (MapLifetimeScope와 동일)
```

#### 옵션 / 파라미터

| 파라미터 | 설명 | 기본값 | 비고 |
|----------|------|--------|------|
| `TotalLayers` | 맵 총 층 수 | `15` | 최소 4 이상 권장 (보스+캠핑+일반 확보) |
| `MaxNodesPerLayer` | 층당 최대 노드 수 | `6` | 99 이하 필수 (Id = Layer×100+Column 구조) |
| `EliteMinLayer` | Elite 등장 최소 층 | `5` | 0층부터 세므로 6번째 층 이후 |
| `MinCampingCount` | 캠핑 최소 보장 수 | `2` | 보스 직전층 제외 기준 |

#### 주의 사항

> ⚠️ 사용 시 주의해야 할 사항을 기록합니다.

- `MapRepository`가 `Application.persistentDataPath/map_save.json`에 저장하므로, 새 게임 시작 시 반드시 `IMapRepository.Delete()`를 호출해 이전 저장 데이터를 삭제해야 한다.
- `MapData` 불러오기 후 `BuildLookups()`를 빠뜨리면 `NodeById`, `NodesByLayer`가 null이어서 즉시 NullReferenceException이 발생한다. `MapRepository.Load()` 내부에서 자동 처리하므로 직접 `JsonUtility.FromJson`을 쓰지 않는다.
- `MaxNodesPerLayer`를 100 이상으로 설정하면 `MapNode.Id = Layer * 100 + Column` 계산에서 Id 충돌이 발생한다.

### 🛠️ 스테이지 완료 후 맵 복귀 연결 가이드

#### 개요
- **대상 :** 각 스테이지 씬 담당자
- **목적 :** 스테이지 종료 시 `StageLoader.NotifyStageCompleted()`를 호출해 맵으로 복귀하는 방법 안내

#### 사용 방법

**Step 1.** 스테이지 씬의 완료 로직 클래스에서 `IStageLoader`를 주입받는다.
```csharp
private IStageLoader _stageLoader;

[Inject]
public void Construct(IStageLoader stageLoader)
{
    _stageLoader = stageLoader;
}
```

**Step 2.** 전투 승리, 이벤트 선택 완료 등 스테이지가 끝나는 시점에 결과를 만들어 전달한다.
```csharp
private void OnStageEnd(bool isVictory)
{
    StageResult result = new StageResult
    {
        IsSuccess = isVictory,
        IsGameOver = isVictory == false,
        RewardIds = new List<string> { "reward_001" }
    };

    // StageLoader는 IStageLoader 인터페이스에 NotifyStageCompleted가 없으므로
    // 현재는 캐스팅해서 호출 (추후 인터페이스에 노출 예정)
    (_stageLoader as StageLoader)?.NotifyStageCompleted(result);
}
```

#### 주의 사항

> ⚠️ 사용 시 주의해야 할 사항을 기록합니다.

- `NotifyStageCompleted()`가 현재 `IStageLoader` 인터페이스에 노출되어 있지 않다. 추후 인터페이스에 추가하거나, 별도 이벤트 버스를 통한 처리로 개선 예정 (refactoring.md 이슈 #3 참고).
