### 📌 이슈 #1 : SkillManager — LINQ·ToList 가비지 발생

#### 문제 사항
- **문제 설명 :** `GetSkillsByType()`에서 매 호출마다 LINQ iterator, 람다 클로저, `List<SkillData>`가 새로 생성됐다. `GetAllSkills()`도 `_skillDict.Values.ToList()`로 매번 리스트를 할당했다. `TickCoolTimes()`의 `Keys.ToList()`도 마찬가지였다.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | `GetSkillsByType`: LINQ Where + ToList → 매 호출 3회 할당. `GetAllSkills`: ToList → 매 호출 1회 할당. `TickCoolTimes`: Keys.ToList() → 매 호출 1회 할당. |
| **After** | 생성자에서 `_skillsByType`, `_allSkillsList` 미리 구성. `TickCoolTimes`는 `List<int> _keyBuffer` 재사용. 조회 시 신규 할당 0. |

```csharp
// Before
public IReadOnlyList<SkillData> GetSkillsByType(SkillType type)
    => _skillDict.Values.Where(s => s.Type == type).ToList(); // iterator + 클로저 + List 할당

public void TickCoolTimes(BaseCharacter character)
{
    foreach (var key in coolTimes.Keys.ToList()) // 매 호출 List 할당
        if (coolTimes[key] > 0) coolTimes[key]--;
}

// After — 생성자에서 미리 구성
private readonly Dictionary<SkillType, List<SkillData>> _skillsByType;
private readonly List<SkillData> _allSkillsList;
private readonly List<string> _keyBuffer = new List<string>();

public IReadOnlyList<SkillData> GetSkillsByType(SkillType type)
{
    if (_skillsByType.TryGetValue(type, out List<SkillData> list)) return list;
    return _emptyList; // 할당 없음
}

public void TickCoolTimes(BaseCharacter character)
{
    _keyBuffer.Clear();
    foreach (string key in coolTimes.Keys) _keyBuffer.Add(key); // 버퍼 재사용
    foreach (string key in _keyBuffer)
        if (coolTimes[key] > 0) coolTimes[key]--;
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** SkillData는 초기화 이후 변하지 않으므로 생성자에서 모든 캐시를 구성하면 조회 시 할당이 전혀 없다. 배틀 루프에서 매 턴 호출될 수 있는 메서드이므로 GC 압박을 최소화하는 것이 중요하다.
- **고려했던 다른 방법 :** `ListPool<T>` (UnityEngine.Pool) 활용
- **최종 선택 근거 :** 데이터가 불변이므로 풀링보다 단순 캐싱이 더 적합하다. `_keyBuffer`는 싱글턴 단위로 하나만 유지하므로 메모리 낭비도 없다.

### 📌 이슈 #2 : SkillId 타입 string → int 시도 후 string으로 복귀

#### 문제 사항
- **문제 설명 :** `SkillId`를 `int`로 변경했으나, 시트 ID가 앞자리 `0`을 포함하는 8자리 포맷(`01010000`)이라 `int` 변환 시 `1010000`으로 표현되어 원본 포맷 정보가 손실된다. 파서·에셋 이름·디버그 출력 등 문자열 ID를 그대로 다루는 곳이 많아 변환 비용이 오히려 더 크다고 판단해 `string`으로 복귀했다.

#### Before / After

| 구분 | 내용 |
|------|------|
| **시도 (int)** | `SkillId: int`, `Dictionary<int, SkillData>`, `Dictionary<int, int>` (쿨타임) |
| **최종 (string 복귀)** | `SkillId: string`, `Dictionary<string, SkillData>`, `Dictionary<string, int>` (쿨타임) |

```csharp
// int 시도 (롤백됨)
public SkillData GetSkill(int skillId) { ... }
void RegisterCoolTime(BaseCharacter character, int skillId, int coolTime);

// 최종 — string 복귀
public SkillData GetSkill(string skillId) { ... }
void RegisterCoolTime(BaseCharacter character, string skillId, int coolTime);
```

**변경된 파일:** `SkillData.cs`, `ISkillManager.cs`, `SkillManager.cs`, `SkillBase.cs`

#### 주요 성과 및 해결 이유 (Why)

- **int 시도 이유 :** Dictionary 해시 계산이 단순하고 값 타입이라 힙 참조가 없다.
- **string으로 복귀한 이유 :** 시트 ID 포맷(`01010000`)이 앞자리 0 포함 8자리 문자열이다. `int` 변환 시 `1010000`이 되어 포맷 불일치가 생기고, 파서·에셋명·로그 등 문자열을 그대로 쓰는 경로에서 매번 변환이 필요해진다. 실질적인 성능 이점보다 유지보수 비용이 더 크다.
- **최종 선택 근거 :** ID 원본 포맷을 그대로 유지하는 `string`이 이 프로젝트 ID 구조에 더 적합하다.

### 📌 이슈 #3 : SkillData.SetData() — 인자 13개 문제

#### 문제 사항
- **문제 설명 :** 파서 지원을 위해 추가한 `SetData()`의 인자가 13개로 너무 많아 호출부 가독성이 나빴다.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | `SetData(int skillId, string skillName, string description, SkillType type, ...)` — 인자 13개 나열 |
| **After** | `SetData(SkillRawData raw)` — 파라미터 오브젝트 패턴으로 단일 인자 |

```csharp
// Before
skillData.SetData(1010000, "기본 공격", "설명", SkillType.OFFENSIVE, SkillTarget.Enemy,
    10f, 0, 0, 0, 3, 12, 0, "0100");

// After
SkillRawData raw = new SkillRawData
{
    SkillId = "01010000",
    SkillName = "기본 공격",
    Type = SkillType.OFFENSIVE,
    Target = SkillTarget.Enemy,
    Value = 10f,
    UseableSlots = 3,
    EnemyTargetMask = 12,
    IconAddress = "0100"
};
skillData.SetData(raw);

// SkillRawData 구조체
public struct SkillRawData
{
    public string SkillId;
    public string SkillName;
    public string Description;
    public SkillType Type;
    public SkillTarget Target;
    public float Value;
    public int CoolTime;
    public int Duration;
    public int UseableClasses;
    public int UseableSlots;
    public int EnemyTargetMask;
    public int AllyTargetMask;
    public string IconAddress;
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** 13개 필드 모두 TSV에서 오는 필수값이라 선택적 필드가 없다. 선택 필드가 많을 때 빛나는 빌더 패턴보다 파라미터 오브젝트(struct)가 더 단순하고 적합하다.
- **고려했던 다른 방법 :** 빌더 패턴 (`SkillData.Builder().WithId().WithType()...Build()`)
- **최종 선택 근거 :** 빌더는 구현 복잡도 대비 이점이 없다. struct는 힙 할당도 없고(값 타입), 파서가 한 행을 읽어 채우는 용도에 딱 맞는다.

### 📌 이슈 #4 : TurnManager.StartBattleAsync() 중복 호출 방지

#### 문제 사항
- **문제 설명 :** `StartBattleAsync()`가 배틀 진행 중에 다시 호출되면 `_turnCount`가 0으로 리셋되고 두 개의 배틀 루프가 동시에 돌아가는 문제가 생길 수 있었다.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | 중복 호출 시 `_turnCount` 리셋 및 루프 이중 실행 가능 |
| **After** | 진입 시 `_isBattleActive` 체크 → 이미 실행 중이면 경고 후 즉시 반환 |

```csharp
// Before
public async UniTask StartBattleAsync(CancellationToken token = default)
{
    _turnCount = 0;
    _isBattleActive = true;
    // ...
}

// After
public async UniTask StartBattleAsync(CancellationToken token = default)
{
    if (_isBattleActive)
    {
        Debug.LogWarning("[TurnManager] 이미 배틀이 진행 중입니다.");
        return;
    }

    _turnCount = 0;
    _isBattleActive = true;
    // ...
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** `_isBattleActive`는 이미 배틀 진행 상태를 나타내는 플래그로 존재했으므로 별도 락(lock) 없이 단순 가드로 충분하다. TurnManager는 싱글턴이고 Unity 메인 스레드에서만 호출되므로 동시성 문제가 없다.
- **고려했던 다른 방법 :** `CancellationTokenSource` 재발급으로 기존 루프 취소 후 재시작
- **최종 선택 근거 :** 중복 실행 자체를 막는 것이 의도에 맞다. 재시작이 필요한 경우는 `BattleEnd()` 호출 후 새로 `StartBattleAsync()`를 부르는 흐름으로 처리한다.

### 🛠️ SkillManager 조회 가이드

#### 개요
- **대상 :** SkillManager를 주입받는 모든 클래스
- **목적 :** 스킬 데이터 조회 및 쿨타임 관리 API 사용법 안내

#### 사용 방법

**Step 1.** VContainer로 `ISkillManager` 주입받기
```csharp
private readonly ISkillManager _skillManager;
public MyClass(ISkillManager skillManager) { _skillManager = skillManager; }
```

**Step 2.** ID로 단건 조회
```csharp
SkillData skill = _skillManager.GetSkill("01010000");
```

**Step 3.** 타입별 조회 (할당 없음 — 캐시 반환)
```csharp
IReadOnlyList<SkillData> offensiveSkills = _skillManager.GetSkillsByType(SkillType.OFFENSIVE);
```

**Step 4.** 스킬 사용 후 쿨타임 등록
```csharp
_skillManager.RegisterCoolTime(character, skill.SkillId, skill.CoolTime);
```

**Step 5.** 남은 쿨타임 확인
```csharp
int remaining = _skillManager.GetRemainingCoolTime(character, skill.SkillId);
bool canUse = remaining == 0;
```

#### 주의 사항

> ⚠️ 사용 시 주의해야 할 사항을 기록합니다.

- `GetSkillsByType()` / `GetAllSkills()` 반환값은 내부 캐시의 참조다. 반환된 리스트를 외부에서 수정하지 않는다.
- `TickCoolTimes()`는 TurnManager가 라운드 종료 시 자동 호출하므로 별도 호출 불필요.

### 🛠️ SkillData 파서 연동 가이드

#### 개요
- **대상 :** 추후 TSV/CSV 파서 구현 담당자
- **목적 :** 코드에서 SkillData를 프로그래밍 방식으로 생성하는 방법 안내

#### 사용 방법

**Step 1.** `ScriptableObject.CreateInstance<SkillData>()`로 인스턴스 생성

**Step 2.** `SkillRawData` 구조체에 시트 값 할당

**Step 3.** `skillData.SetData(raw)` 호출

```csharp
SkillRawData raw = new SkillRawData
{
    SkillId         = "01010000",
    SkillName       = "기본 공격",
    Type            = SkillType.OFFENSIVE,
    Target          = SkillTarget.Enemy,
    Value           = 10f,
    CoolTime        = 0,
    Duration        = 0,
    UseableClasses  = 0,         // 0 = 전체 허용
    UseableSlots    = 3,         // 0b0011
    EnemyTargetMask = 12,        // 0b1100
    AllyTargetMask  = 0,
    IconAddress     = "0100"
};

SkillData skill = ScriptableObject.CreateInstance<SkillData>();
skill.SetData(raw);
```

#### 주의 사항

> ⚠️ 사용 시 주의해야 할 사항을 기록합니다.

- `SkillId`가 null 또는 빈 문자열이면 SkillManager 등록에서 제외된다.
- `SkillRawData`는 struct(값 타입)이므로 힙 할당 없이 사용 가능하다.
