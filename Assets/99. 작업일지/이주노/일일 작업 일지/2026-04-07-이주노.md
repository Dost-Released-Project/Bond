### 📌 이슈 #1 : SkillData — 스킬 시트 데이터를 코드에서 관리하는 방법

#### 문제 사항
- **문제 설명 :** 스킬 시트(TSV)에 ID, 타입, 수치, 비트마스크(사용 가능 칸/진영) 등 다양한 필드가 정의되어 있으나, 이를 Unity에서 관리할 구조가 없었다. 스킬 데이터를 코드에 하드코딩하면 수정·확장이 어렵다.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | SkillBase에 damage, currentLevel만 존재. 스킬 데이터 구조 없음. |
| **After** | SkillData ScriptableObject로 ID/타입/수치/비트마스크/아이콘 주소 전 필드 보관. |

```csharp
// Before — SkillBase.cs
public abstract class SkillBase : MonoBehaviour
{
    public float SkillCooldown { get; private set; }
    protected float damage;
    protected int currentLevel;
    public void Init() { }
    public abstract void UseSkill();
}

// After — SkillData.cs (신규)
[CreateAssetMenu(menuName = "Bond/SkillSystem/SkillData", fileName = "SK_00000000")]
public class SkillData : ScriptableObject
{
    [SerializeField] private string _skillId;
    [SerializeField] private SkillType _type;
    [SerializeField] private float _value;
    [SerializeField] private int _coolTime;
    [SerializeField] private int _duration;
    [SerializeField] private int _useableClasses;
    [SerializeField] private int _useableSlots;
    [SerializeField] private int _enemyTargetMask;
    [SerializeField] private int _allyTargetMask;
    [SerializeField] private string _iconAddress;
    // ... 프로퍼티
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** ScriptableObject를 선택한 이유는 에디터에서 직접 값을 입력/확인할 수 있고, Addressables 비동기 로드와도 자연스럽게 연결되기 때문이다. 비트마스크 해석 로직은 다른 팀원의 매니저가 담당하기로 결정되어 SkillData는 순수 데이터 컨테이너로만 유지했다.
- **고려했던 다른 방법 :** CSV/TSV 런타임 파싱, JSON 파일 로드
- **최종 선택 근거 :** ScriptableObject는 Unity 에디터 인스펙터와 직접 통합되며, 에셋 단위로 관리되어 팀 협업 시 충돌이 적다.

### 📌 이슈 #2 : SkillManager — 스킬 데이터 접근 창구 및 쿨타임 관리

#### 문제 사항
- **문제 설명 :** 스킬 데이터를 각 클래스가 직접 참조하면 의존성이 분산되고, 쿨타임 상태도 어디서 관리해야 하는지 불명확했다. VContainer DI를 활용해 단일 창구를 만들 필요가 있었다.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | SkillManager 없음. 스킬 데이터 접근 경로 미정의. TurnManager가 쿨타임을 알지 못함. |
| **After** | ISkillManager 인터페이스 + SkillManager 구현체로 ID 조회, 타입 조회, 쿨타임 등록/감소/조회 제공. TurnManager가 라운드 종료 시 자동으로 TickCoolTimes() 호출. |

```csharp
// Before — TurnManager 생성자
public TurnManager()
{
    Debug.Log($"[TurnManager] 생성됨!");
}

// After — ISkillManager 주입 + 라운드 종료 시 쿨타임 감소
public TurnManager(ISkillManager skillManager)
{
    _skillManager = skillManager;
}

// PlayRoundAsync() 끝에 추가
foreach (var unit in _units)
{
    if (unit is BaseCharacter character)
        _skillManager.TickCoolTimes(character);
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** ISkillManager 인터페이스를 두어 SkillManager 구현체와 사용처를 분리했다. 덕분에 추후 구현체를 교체하거나 Mock으로 대체해도 사용 측 코드를 수정할 필요가 없다.
- **고려했던 다른 방법 :** SkillManager를 MonoBehaviour Singleton으로 구현
- **최종 선택 근거 :** 기존 프로젝트가 VContainer DI 패턴을 일관되게 사용 중이므로 동일한 방식으로 통합하여 코드 일관성을 유지했다.

### 📌 이슈 #3 : SkillBase — SkillData 바인딩 방식

#### 문제 사항
- **문제 설명 :** SkillBase가 스킬 데이터와 연결되지 않아 UseSkill() 구현 시 수치 참조 방법이 없었다.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | `Init()` 빈 메서드. SkillData 참조 없음. |
| **After** | `Init(string skillId, ISkillManager skillManager)`로 호출 측에서 ISkillManager를 직접 전달. SkillCooldown도 SkillData.CoolTime으로 자동 설정. |

```csharp
// Before
public void Init() { }

// After
protected SkillData _skillData;

public void Init(string skillId, ISkillManager skillManager)
{
    _skillData = skillManager.GetSkill(skillId);
    if (_skillData != null)
        SkillCooldown = _skillData.CoolTime;
}
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** SkillBase는 MonoBehaviour이므로 VContainer에 등록되지 않은 경우 `[Inject]` 필드 주입이 동작하지 않아 null 참조 위험이 있다. 대신 Init() 호출 측(이미 ISkillManager를 주입받은 클래스)에서 인자로 넘기는 방식을 선택했다.
- **고려했던 다른 방법 :** `[Inject]` 필드 주입 / Inspector에서 SkillData를 직접 SerializeField로 할당
- **최종 선택 근거 :** MonoBehaviour는 VContainer 등록 여부가 불확실하므로 인자 전달 방식이 가장 안전하다. ID 기반 조회로 SkillManager가 단일 진실 공급원(Single Source of Truth) 역할을 유지한다.

### 📌 이슈 #4 : SkillBase [Inject] 필드 주입 — MonoBehaviour 주의점

#### 문제 사항
- **문제 설명 :** 초안에서 SkillBase에 `[Inject] protected ISkillManager _skillManager`를 선언했으나, MonoBehaviour는 VContainer에 명시적으로 등록된 경우에만 `[Inject]`가 동작한다. 씬에 직접 배치하거나 AddComponent로 생성된 MonoBehaviour는 주입이 실행되지 않아 null 참조 에러가 발생할 수 있다.

#### Before / After

| 구분 | 내용 |
|------|------|
| **Before** | SkillBase에 `[Inject]` 필드 선언. VContainer 미등록 시 _skillManager가 null. |
| **After** | `[Inject]` 필드 제거. `Init(string skillId, ISkillManager skillManager)` 인자 전달 방식으로 변경. |

```csharp
// Before — VContainer 미등록 시 null 위험
[Inject] protected ISkillManager _skillManager;

public void Init(string skillId)
{
    _skillData = _skillManager.GetSkill(skillId); // null 참조 가능
}

// After — 호출 측에서 주입받은 skillManager를 직접 전달
public void Init(string skillId, ISkillManager skillManager)
{
    _skillData = skillManager.GetSkill(skillId); // 안전
}

// 사용 예시 (ISkillManager를 이미 주입받은 클래스 내부)
mySkill.Init("01010000", _skillManager);
```

#### 주요 성과 및 해결 이유 (Why)

- **선택 이유 :** MonoBehaviour의 `[Inject]`는 VContainer 등록이 전제되어야 하지만, 스킬 오브젝트가 항상 DI 컨테이너를 통해 생성된다고 보장할 수 없다.
- **고려했던 다른 방법 :** 모든 SkillBase 하위 클래스를 `builder.RegisterComponentInHierarchy<>()`로 VContainer에 등록
- **최종 선택 근거 :** 등록 강제는 사용 측에 제약이 생긴다. 인자 전달 방식은 호출 측만 ISkillManager를 갖고 있으면 되므로 더 유연하고 안전하다.

### 🛠️ SkillData 에셋 생성 가이드

#### 개요
- **대상 :** 스킬 시트에 정의된 각 스킬 1행
- **목적 :** 스킬 데이터를 ScriptableObject 에셋으로 생성하여 SkillManager에 등록

#### 사전 준비

```
Assets/04. SkillData/ 폴더 생성 (없는 경우)
```

#### 사용 방법

**Step 1.** Project 창에서 우클릭 → `Create → Bond/SkillSystem/SkillData`

**Step 2.** 파일명을 `SK_[ID]` 형식으로 지정 (예: `SK_01010000`)

**Step 3.** Inspector에서 스킬 시트 데이터를 입력

```
SkillId       : 01010000
SkillName     : (스킬 이름)
Type          : OFFENSIVE
Target        : Enemy
Value         : 10
CoolTime      : 0
Duration      : 0
UseableClasses: 0  (0 = 모든 클래스)
UseableSlots  : 3  (0b0011 = 슬롯 0,1)
EnemyTargetMask: 12 (0b1100 = 적 슬롯 2,3)
AllyTargetMask : 0
IconAddress   : 0100
```

#### 옵션 / 파라미터

| 필드 | 설명 | 기본값 | 비고 |
|------|------|--------|------|
| `SkillId` | 8자리 ID 문자열 | `""` | 시트 ID 그대로 입력 |
| `UseableClasses` | 클래스 비트마스크 | `0` | 0 = 전체 허용 |
| `UseableSlots` | 사용 가능 칸 비트마스크 | `0` | 아군 기준 `[3][2][1][0]` |
| `EnemyTargetMask` | 적 진영 타겟 비트마스크 | `0` | 적 기준 `[0][1][2][3]` |
| `AllyTargetMask` | 아군 진영 타겟 비트마스크 | `0` | 아군 기준 `[3][2][1][0]` |

#### 주의 사항

> ⚠️ 사용 시 주의해야 할 사항을 기록합니다.

- `SkillId`가 비어 있으면 SkillManager 초기화 시 경고 로그가 출력되고 등록에서 제외된다.
- 비트마스크 해석(슬롯 매핑)은 SkillData/SkillManager가 아닌 다른 팀원 담당 매니저에서 처리한다.

### 🛠️ SkillScope 씬 설정 가이드

#### 개요
- **대상 :** 씬 담당자
- **목적 :** SkillManager를 VContainer Singleton으로 등록하여 TurnManager, SkillBase 등에 주입

#### 사용 방법

**Step 1.** 씬에 빈 GameObject 생성 → 이름: `SkillScope`

**Step 2.** `SkillScope` 컴포넌트 추가 → Inspector의 `_allSkillData` 배열에 모든 SkillData 에셋 할당

**Step 3.** `TurnLifetimeScope` 오브젝트 선택 → Inspector의 `Parent` 필드에 위에서 만든 `SkillScope` 오브젝트 지정

```
씬 계층 예시:
SkillScope          ← SkillManager 등록
  └─ TurnLifetimeScope  ← TurnManager(ISkillManager 주입받음) 등록
```

#### 주의 사항

> ⚠️ 사용 시 주의해야 할 사항을 기록합니다.

- `TurnLifetimeScope`의 Parent를 `SkillScope`로 지정하지 않으면 VContainer가 `ISkillManager`를 resolve하지 못해 런타임 에러가 발생한다.
- SkillBase는 `[Inject]` 방식 대신 `Init(skillId, skillManager)` 인자 전달 방식을 사용하므로, SkillBase 하위 클래스를 VContainer에 별도 등록할 필요 없다.

### 🔁 이슈 #2 재현 방법 : TurnManager ISkillManager 미주입 시 에러

#### 환경 정보

| 항목 | 내용 |
|------|------|
| OS | Windows 11 |
| 런타임 / 버전 | Unity 6000.3.10f1 |
| 관련 라이브러리 버전 | VContainer 1.17.0 |
| 기타 환경 정보 | SkillScope의 Parent 미설정 상태 |

#### 재현 절차

1. 씬에 `SkillScope` 오브젝트를 추가하지 않거나, `TurnLifetimeScope`의 Parent를 지정하지 않는다.
2. Play 모드 진입
3. VContainer가 `TurnManager` 생성 시 `ISkillManager`를 resolve하지 못해 예외 발생

```
VContainer.VContainerException:
  Unable to resolve ISkillManager
```

#### 예상 결과 (재현 성공 시)

```
VContainerException: Unable to resolve ISkillManager
```
